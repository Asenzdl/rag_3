# 领航员文档模板

> **适用范围**：技术学习文档——领航员型
> **核心使命**：串起代码蓝图的全局脉络，让读者从"看到一棵棵树"变成"看到森林"
> **关联规范**：`project_info/task_doc_design_spec.md`（design.md 规范）、`anchor_doc_template.md`（锚点文档模板）

---

## 〇、核心理念

### 代码蓝图的领航员

design.md 的代码蓝图是超高密度的信息体——每个函数的步骤注释都写得很细，但读者按模块逐个读下去，只看到一棵棵树。领航员文档串起全局脉络，让读者在深入细节之前先看到全貌：各模块之间怎么协作、数据怎么流转、整体设计意图是什么。

### 决策是路标，不是目的地

领航员在串脉络的过程中，走到某个协作节点会自然冒出决策——"这里用了工厂函数而不是直接构造"。领航员点到为止，指出衔接动机，深度展开交给锚点文档。决策是旅程中的路标，不是旅程本身。

### 全局视角的失效

领航员不只串脉络，还要指出这个整体架构在什么条件下会出问题——不是某个技术的失效，而是模块协作的断裂点、设计原则的适用范围。

---

## 一、领航员文档模板（3+2 结构）

> 路径：`.project/docs/task_x.x/[task_x.x]_navigator.md`
> 定位：串起代码蓝图全局脉络的导航文档，通常一个 Task 一篇

~~~markdown
# [Task 主题] — 设计导航

> **关联 Task**：Task X.X [任务名称]
> **文档类型**：技术学习文档（领航员）
> **关联文档**：[本 Task 的锚点文档列表，领航员点到为止的决策在这些文档中深度展开]
> **覆盖的 Task 知识点**：[声明本篇覆盖的非决策类全局知识点]

---

## 一、设计全景

[这个 Task 整体在做什么？围绕什么核心设计原则？关键模块有哪些？]

> **定位**：鸟瞰图，不是逐个模块介绍。读者读完本章应能用一句话概括整个 Task 的设计意图。
> **核心设计原则**：必须点出贯穿整个 Task 的设计原则（如"调用方只依赖抽象，不依赖具体"），
> 这个原则会把散落的模块和决策串成一条线。
> **关键模块速览**：每个模块用一句话说清职责和存在理由，不展开内部逻辑。
> **与 design.md 的区别**：design.md 的"模块结构"是文件组织 + 职责边界，
> 本章是"为什么这样组织"——设计意图层面的全局视角。

### 核心设计原则
[一条原则，把散落的模块和决策串起来]

### 模块地图
[每个模块一句话：职责 + 存在理由]

---

## 二、协作脉络

[模块之间怎么协作？数据怎么流转？控制权怎么传递？]

> **定位**：领航员最核心的章节。读者最需要的"全局视角"在这里。
> **Mermaid 图**：本章必须包含至少一个 Mermaid 图，展示模块间的协作关系或数据流。
> **决策的路标**：走到某个协作节点时，自然引出决策——"这里用了 X 而不是 Y，因为……"。
> 点到为止，指出衔接动机即可，深度展开交给锚点文档（用"详见 xxx.md"引用）。
> **渐进式呈现**：从主流程开始，逐步展开分支和异常路径，不要一开始就铺开所有细节。

### 主流程
[核心数据流 / 控制流，配合 Mermaid 图]

### 分支与异常
[偏离主流程的路径，什么时候触发、怎么处理、怎么回到主流程]

### 决策路标
[协作过程中冒出的决策，点到为止 + 引用锚点文档]

---

## 三、边界与失效

[这个整体架构在什么条件下会出问题？]

> **与锚点文档"失效边界"的区别**：锚点文档讲某个技术的失效边界，
> 本章讲这个整体架构的失效边界——模块协作的断裂点、设计原则的适用范围。
> **至少 2 种失效模式**，每种包含完整的"症状→原因→诊断→解决"链。

### 3.1 [架构级失效模式 1]
- **症状**：[出问题时表现为什么——通常是跨模块的异常表现]
- **原因**：[为什么会出现这个问题——通常是设计原则的适用边界被突破]
- **诊断方法**：[如何发现/确认，尽量可量化]
- **解决方案**：[如何修复或规避]

### 3.2 [架构级失效模式 2]
- **症状**：...
- **原因**：...
- **诊断方法**：...
- **解决方案**：...

## 附录 A、代码映射

> **定位**：模块职责与代码位置的对照表，服务于"想看实现去哪找"。
> **与 design.md 的区别**：design.md 的"模块结构"描述文件组织，

| 模块 | 对应代码位置 | 职责 |
|-----|------------|------|
| [模块1] | `src/xxx/yyy.py:ClassName` | [一句话职责] |
| [模块2] | `src/xxx/yyy.py:function_name` | [一句话职责] |

## 附录 B、知识地图

> **定位**：围绕本 Task 的整体主题，给出一个生产级程序员应掌握的完整知识体系。
> 领航员覆盖了全局脉络，但每个模块的深层知识需要锚点文档和外部资源补充。

### 本文覆盖范围

| [领域2：设计原则] | [深度] | [章节名] |

### 延伸学习路径

| [方向1] | [关键词、论文、文档链接] | [需先掌握什么] | [为什么这个方向值得深入] |
| [方向2] | ... | ... | ... |
~~~

## 二、行间接引机制

与锚点文档一致。叙事过程中，写到某个点必须收住，就在收住处留一句话级别的指引。

领航员文档中，行间接引最常见的触发场景是：
- 协作脉络中点到某个决策，但展开会打断流程 → 引用锚点文档 + 给出方向关键词
- 提到某个模块的内部原理，但深入它是锚点文档的事 → 指出方向关键词

## 三、深度标准

### 深度层级

| 层级 | 描述 | 示例 |
|------|------|------|
| L1 浅层 | 告诉你"有什么模块" | "项目有 config、factory、retriever 三个模块" |
| L2 中层 | 告诉你"模块为什么这样协作" | "factory 创建 retriever，config 驱动 factory 的行为——三者形成配置驱动的创建链" |
| L3 深层 | 告诉你"协作的深层逻辑是什么"、"什么条件下协作会断裂" | "配置驱动的创建链隐含了一个前提：所有组件都可以无状态构造。一旦某个组件需要运行时状态（如连接池），factory 模式就需要重构为 registry 模式" |

**目标**：协作脉络至少达到 L2，至少一个章节达到 L3。

### 文档长度指引

| 维度 | 建议 |
|------|------|
| 全文建议长度 | 1000~2500 字（不含代码块和 Mermaid 图） |
| 超出上限时 | 检查是否越界展开了某个决策的深度——应该交给锚点文档 |

### 量化数据来源标注规范

与锚点文档一致：`[实测]` / `[计算]` / `[估算]` / `[引用:来源]`。

### 必选深度要素

| 要素 | 要求 | 适用条件 |
|------|------|--------|
| 核心设计原则 | 点出贯穿 Task 的设计原则 | 所有领航员文档 |
| 协作脉络 | 模块间协作关系 + 数据流，Mermaid 图 | 所有领航员文档 |
| 决策路标 | 点到为止 + 引用锚点文档 | 有架构决策的 Task |
| 架构级失效 | 至少 2 种：症状→原因→诊断→解决 | 所有领航员文档 |
| 代码映射 | 模块→代码位置的映射表 | 所有领航员文档 |
| 知识地图 | 覆盖范围 + 延伸学习路径 | 所有领航员文档 |

### 自检问题

写完领航员文档后，必须逐条回答：

1. **全景测试**：读者读完能否用一句话概括整个 Task 的设计意图？如果不能 → 设计全景太浅或缺少核心设计原则
2. **脉络测试**：读者能否说清各模块之间怎么协作、数据怎么流转？如果不能 → 协作脉络不够清晰
3. **诊断测试**：如果这个架构在生产环境出了跨模块的问题，读者能否根据文档定位原因？如果不能 → 缺乏架构级失效或诊断方法不具体
4. **决策测试**：领航员中提到的决策，是否都引用了锚点文档？如果有决策展开过多 → 抢了锚点文档的活
5. **映射测试**：读者能否根据文档快速找到每个模块的代码位置？如果不能 → 缺乏代码映射
6. **接引测试**：读者读完后，是否知道围绕本 Task 还有哪些知识体系需要深入？如果不知道 → 缺乏知识地图

## 四、与锚点文档的分工

| 维度 | 领航员文档 | 锚点文档 |
|------|-----------|---------|
| 使命 | 串全局脉络 | 展开一个知识领域 |
| 叙事重心 | 模块间的**关系** | 技术本身的**深度** |
| 决策怎么处理 | 路标——点到为止，指出衔接动机 | 锚点——深度展开，体系化讲解 |
| 失效视角 | 架构级——模块协作的断裂点 | 技术级——某个技术的失效边界 |
| 章节数 | 3 章 + 2 附录 | 5 章 + 2 附录 |
| 长度 | 1000~2500 字 | 2000~4000 字 |

### 读者阅读顺序建议

1. 先读领航员 → 获得全局视角
2. 再读感兴趣的锚点文档 → 深入特定领域
3. 需要编码时回到 design.md 的代码蓝图 → 获得施工细节

## 五、生成流程

### 规划轮

领航员文档和锚点文档在同一轮规划中确定：

```markdown
## 技术文档规划

| # | 文件名 | 类型 | 主题 | 叙事线 | 锚定决策 | 覆盖的 Task 知识点 | L3 聚焦 |
|---|--------|------|------|--------|---------|-------------------|---------|
| 1 | task_x.x_navigator.md | 领航员 | ... | 全景→协作→边界 | 全部决策 | [全局性知识点] | [协作脉络] |
| 2 | xxx.md | 锚点 | ... | 问题→原理→全景→实践→边界 | 决策1, 决策3 | [非决策类知识点] | [具体章节] |
| 3 | yyy.md | 锚点 | ... | 问题→原理→全景→实践→边界 | 决策2 | [非决策类知识点] | [具体章节] |
```

### 生成顺序

1. **先生成领航员文档**——因为领航员需要全局视角，此时设计决策的记忆最清晰
2. **再逐篇生成锚点文档**——领航员中的决策路标已经为锚点文档划定了边界
3. 每篇生成后立即执行深度自检
4. 全部完成 → 全量复查 + 跨文档去重检查

---

## 六、质量检查清单

### 叙事章节必检项

- [ ] 明确标注关联 Task、覆盖的 Task 知识点
- [ ] 设计全景有点出的核心设计原则 + 模块一句话速览
- [ ] 协作脉络清晰，有 Mermaid 图展示数据流/模块协作
- [ ] 决策路标点到为止，深度展开引用锚点文档
- [ ] 边界与失效至少 2 种架构级失效，四要素完整
- [ ] 行间接引在叙事收住处自然出现

### 附录章节检查项

- [ ] 代码映射表完整（模块→代码位置）
- [ ] 知识地图包含覆盖范围和延伸学习路径

### 跨文档与边界检查项

- [ ] 关联文档已标注锚点文档链接
- [ ] 决策展开程度与锚点文档不重复
- [ ] 深度自检通过：全景/脉络/诊断/决策/映射/接引

## 设计思考

### 为什么领航员是 3 章而不是 5 章

领航员的使命是"串全局脉络"，不需要从"核心问题"讲起——读者刚读完 design.md，知道问题是什么。

1. 你站在哪（设计全景）
2. 各部分怎么连（协作脉络）
