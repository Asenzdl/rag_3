# 锚点文档模板

> **适用范围**：技术学习文档——锚点型
> **核心使命**：从一个架构决策出发，展开一个知识领域，讲透这个领域本身
> **关联规范**：`project_info/task_doc_design_spec.md`（design.md 规范）、`navigator_doc_template.md`（领航员文档模板）

---

## 〇、核心理念

### 决策是锚点，不是边界

架构决策是进入一个知识领域的入口，不是文档的天花板。文档从决策出发，展开该领域的体系化知识——不只回答"项目为什么选这个"，还要回答"这个东西本身是什么、体系是什么"。

### 写到决策讲透就收

文档的深度边界由架构决策的主线决定——主线讲透了就收。但收住处必须给出接引，让读者知道往哪个方向继续深入。

### 深度来自洞察，不来自章节

模板搭好舞台（叙事线清晰、章节精简、边界明确），给 AI 足够的开放空间产出洞察。叙事线规定了节奏，也留白了深度。

---

## 一、锚点文档模板（5+2 结构）

> 路径：`.project/docs/task_x.x/[topic_name].md`
> 定位：围绕一个架构决策的深度教学文档，用渐进式叙事讲透一个知识领域

~~~markdown
# [技术主题]

> **关联 Task**：Task X.X [任务名称]
> **文档类型**：技术学习文档（锚点）
> **锚定决策**：[声明本篇锚定的 design.md 中的决策，如"决策1: 检索策略选型"]
> **覆盖的 Task 知识点**：[声明本篇覆盖的非决策类知识点，如"Liskov 替换原则、Structural subtyping 哲学"]
> **前置知识**：[读者需先了解什么]
> **关联文档**：[同 Task 的领航员文档、其他锚点文档的链接，避免内容重叠]

# 叙事章节（沿"问题→原理→全景→实践→边界"渐进建构）

---

## 一、核心问题

[为什么需要这个技术？解决什么问题？]

> **深度要求**：必须给出具体痛点 + 可感知的示例
> **子问题数量**：建议 2~3 个，每个有具体示例
> **叙事角色**：叙事线的起点——让读者感受到"确实需要这个东西"

### 1.1 [子问题/挑战 1]
[说明 + 具体示例]

### 1.2 [子问题/挑战 2]
[说明 + 具体示例]

---

## 二、技术原理

[详细的算法/原理说明，含公式、图示、流程图]

> **深度要求**：必须解释"为什么有效"（底层机制/数学直觉），不能只说"怎么用"
> **渐进式建构**：算法/配置/Prompt 类主题需展示 V0→V1→V2 演进过程，每步说明改因+效果
> **适用性说明**：工具/框架类主题可用"核心用法 + 进阶技巧"替代渐进式建构
> **量化自然嵌入**：原理解释时，论据须有量化支撑（如复杂度分析、token 预算推算），数据须标注来源
> **Mermaid 图要求**：涉及数据流或 3+ 组件交互的主题，必须至少包含一个 Mermaid 流程图
> **叙事角色**：叙事线的核心——让读者获得"原来如此"的洞察

### 2.1 [核心概念]
[深入解释，不要浅尝辄止]

### 2.2 [关键机制]
[配合代码片段说明]

---

## 三、方案全景

[该领域的方案全貌——有哪些路径？各自优劣？项目选了哪条？]

> **与旧版"方案选型"的区别**：不只是服务于"项目选了哪个"，而是给出该领域的方案全景。
> 读者应能从本章理解该领域的全貌，而不仅仅是项目的选择结论。
> **适用条件**：主题存在 2+ 个业界主流方案时，做完整方案对比；
> 若主题天然缺少有意义的对比空间，可简化为"设计选型说明"。
> **量化自然嵌入**：方案对比时，权衡须有量化维度（如延迟、准确率、扩展成本），数据须标注来源
> **叙事角色**：叙事线的转折——让读者理解"项目选了这个方案"以及"这个领域还有哪些路径"

[解释为什么选当前方案，其他可能方案的不足之处，1~2 段即可]

## 四、实现要点

[用于教学的代码示例（说明原理/用法，不是复制实现代码） + 详细中文注释]

> **深度要求**：
> - 算法/配置/Prompt 类主题：展示从简到繁的演进过程（V0→V1→V2），而非直接贴最终代码
> - 工具/框架类主题：展示核心用法 + 进阶技巧 + 常见配置变体
> **最佳实践融合**：用 ✅/❌ 标注贯穿代码示例，替代独立"最佳实践"章节
> **最小可理解单元原则**：每个代码示例应是一个最小可理解单元——读者无需理解完整项目即可理解该示例说明的原理
> **叙事角色**：叙事线的落地——让读者从理论走向实践

# ✅ 推荐做法 + 为什么这样做
```

```python
# ❌ 常见反模式 + 为什么不应该这样做
```

## 五、失效边界

[什么情况下会翻车？]

> **必选章节**，不可省略。每个技术主题至少覆盖 2 种典型失效场景。
> **四要素完整性要求**：每种失效模式必须包含完整的"症状→原因→诊断方法→解决方案"链。
> 缺少"诊断方法"是最常见的质量问题——"诊断方法"应回答"如何量化/确认问题已发生"。
> **与 design.md 的区别**：design.md 的"常见坑点"是编码时易犯的错，本章节是运行时/生产环境的失效边界。
> **叙事角色**：叙事线的终点——让读者知道技术的边界在哪

- **解决方案**：[如何修复或规避]

- **症状**：...
- **原因**：...
- **诊断方法**：...
~~~

## 二、行间接引机制

叙事过程中，写到某个点必须收住（因为再展开就超出本文范围），就在收住处留一句话级别的指引。

**触发条件**：叙事涉及了某个知识点的表面，但展开它会偏离本文主线。

**写法**：

> [收住处的上下文]——本文只覆盖了 [本文讲了什么层面]，要理解 [更深层的东西]，方向关键词：[关键词]。

**示例**：

> Protocol 的 structural subtyping 与 ABC 的 nominal subtyping 背后是两种类型哲学的分歧——本文只覆盖项目选型所需的层面，要理解这两种哲学在类型系统中的完整演进，方向关键词：Liskov substitution principle / duck typing vs structural typing / PEP 544 设计动机。

## 三、深度标准

### 深度层级

| 层级 | 描述 | 示例 |
|------|------|------|
| L1 浅层 | 告诉你"怎么用" | "MMR 通过 λ 参数平衡相关性和多样性" |
| L2 中层 | 告诉你"为什么这样设计" | "λ=1 退化为纯相似度，λ=0 退化为纯多样性，实际使用 0.5-0.7" |
| L3 深层 | 告诉你"底层机制是什么"、"什么情况下会失败" | "MMR 分数 = λ·sim(q,d) - (1-λ)·max(sim(d,d'))，前半项拉向查询，后半项推离已选文档；当嵌入空间各向异性时，后半项可能选出不相关文档" |

**目标**：每个技术主题的叙事章节至少达到 L2，至少一个章节达到 L3。

### 文档长度指引

| 维度 | 建议 |
|------|------|
| 叙事章节每节下限 | 300~500 字 |
| 全文建议长度 | 2000~4000 字（不含代码块） |
| 超出上限时 | 拆分为多篇锚点文档，每篇聚焦一个子主题 |

### 量化数据来源标注规范

| 标注 | 含义 | 可信度 |
|------|------|--------|
| `[实测]` | 基于实际运行代码的测量结果 | 最高 |
| `[计算]` | 基于理论公式或代码逻辑的推算 | 高 |
| `[估算]` | 基于经验的合理估计 | 中 |
| `[引用:来源]` | 引自外部文献或官方文档 | 取决于来源 |

### 必选深度要素

| 要素 | 要求 | 适用条件 |
|------|------|--------|
| 理论解释 | 不仅"怎么用"，必须解释"为什么有效" | 所有主题 |
| 失效边界 | 至少 2 种：症状→原因→诊断→解决，诊断方法须可量化 | 所有主题 |
| 渐进式建构 | 展示从简到繁的演进（V0→V1→V2） | 算法/配置/Prompt类主题 |
| 核心用法+进阶技巧 | 基础用法 → 进阶配置 → 常见变体 | 工具/框架类主题 |
| 跨方案权衡 | 不仅列选项，必须给出选择指南 | 有方案全景的主题 |
| 代码映射 | 技术概念→代码位置的映射表 | 所有主题 |
| 知识地图 | 覆盖范围 + 延伸学习路径 | 所有主题 |
| Mermaid 流程图 | 涉及数据流或 3+ 组件交互时至少一个 | 数据流/多组件交互主题 |

### 严禁的浅层写法

```markdown
<!-- ❌ 只说结论不说理由 -->
选方案A因为更灵活。

<!-- ✅ 结论+理由+边界条件 -->
选方案A（字典映射）因为：(1) 新增版本只需加一行，O(1) 扩展；
(2) 但牺牲类型安全；(3) 超过10个版本建议迁移方案C。

<!-- ❌ 只说"怎么用"不说"为什么" -->
Few-shot 就是在 prompt 里加示例，LLM 就能学会格式。

<!-- ✅ 解释底层机制 -->
Few-shot 有效的根本原因：Transformer 注意力检索最相似示例并补全输出。
这意味着：(1) 示例相似度决定激活哪个；(2) 顺序影响权重；
(3) 超过注意力头数后边际收益递减。

<!-- ❌ 方案对比只有表格没有选择指南 -->
| 方案 | 优点 | 缺点 |
| A | 简单 | 不灵活 |
| B | 灵活 | 复杂 |

<!-- ✅ 表格+选择指南+量化权衡 -->
| 方案 | 扩展成本 | 类型安全 | 适用规模 |
| A 字典映射 | O(1) | ❌ | ≤10 版本 |
| B 类继承 | O(n) | ✅ | 任意 |
→ 选A：当前仅3个版本，扩展成本优势明显；后续超10版本时迁移C

<!-- ❌ 失效边界只写标题不写诊断链 -->
### 5.1 长上下文失败
Prompt 太长时 LLM 可能忽略中间指令。

<!-- ✅ 失效边界有完整诊断链（四要素缺一不可） -->
### 5.1 长上下文指令淹没
- 症状：LLM 忽略 System Prompt 中的格式指令，输出缺少引用标记
- 原因：注意力机制在长序列上稀释，中间 token 的平均注意力权重不足两端的 30%
- 诊断：对比有无中间 few-shot 时的指令遵从度，差值>40% 即为淹没
- 解决：将关键指令移至消息序列首尾（首因/近因效应），或精简中间内容

<!-- ❌ 量化论据无来源标注 -->
引用格式遵从率从 60% 提升到 90%。

<!-- ✅ 量化论据标注来源 -->
引用格式遵从率从 60% 提升到 90% [估算：基于 20 次手动测试]。
Token 消耗：V2 比 V1 增加约 150 tokens [计算：基于模板字符数估算]。
```

### 自检问题

写完锚点文档后，必须逐条回答：

1. **去用测试**：如果把"怎么用"的部分去掉，文档是否还有独立价值？如果没有 → 缺乏深度
2. **决策测试**：读者能否根据文档理解"为什么项目选了这个方案"？如果不能 → 方案全景太浅
3. **诊断测试**：如果该技术在生产环境出了问题，读者能否根据文档诊断原因？如果不能 → 缺乏失效边界或诊断方法不具体
4. **洞察测试**：文档中是否有至少一个"原来如此"级别的洞察？如果没有 → 可能太浅
5. **映射测试**：读者能否根据文档快速找到项目中的对应代码？如果不能 → 缺乏代码映射
6. **接引测试**：读者读完后，是否知道围绕本文主题还有哪些知识需要深入、往哪个方向走？如果不知道 → 缺乏知识地图或行间接引

## 四、与 design.md 的内容边界

| 知识类型 | design.md 负责 | 锚点文档负责 | 重叠时处理 |
|---------|---------------|-------------|-----------|
| 选型结论 | 本项目选了什么+为什么 | 领域方案全景+通用选择指南 | 锚点文档引用 design.md 的结论，不重复权衡细节 |
| 编码级陷阱 | 常见坑点（写代码时易犯的错） | 失效边界（运行时的失效边界） | 按编码时/运行时分层 |
| 代码组织 | 文件结构+契约速览 | 技术概念→代码位置的语义映射 | 锚点文档不重复文件结构 |
| 异常处理 | 异常捕获表+包装策略 | 技术失效的根因+诊断方法 | design.md 写"怎么捕"，锚点文档写"为什么抛" |
| 底层原理 | 不涉及 | 核心职责——解释"为什么有效" | 锚点文档独有 |

## 五、生成流程

### 规划轮

AI 自主判断每篇锚点文档覆盖哪些设计决策和 Task 知识点，但在规划轮必须声明覆盖范围，确保无遗漏。

```markdown
## 技术文档规划

| # | 文件名 | 类型 | 主题 | 叙事线 | 锚定决策 | 覆盖的 Task 知识点 | L3 聚焦 |
|---|--------|------|------|--------|---------|-------------------|---------|
| 1 | xxx.md | 锚点 | ...  | 问题→原理→全景→实践→边界 | 决策1, 决策3 | [非决策类知识点] | [哪个章节达到 L3] |
| 2 | yyy.md | 领航员 | ...  | 全景→协作→边界 | 全部决策 | [全局性知识点] | [哪个章节达到 L3] |
```

> **遗漏检查**：规划完成后，对照 design.md 的决策列表和 Task 的面试知识点，确认每个决策至少被一篇文档覆盖，每个重要知识点至少被一篇文档触及。

### 逐篇生成规则

1. 先生成 5 个叙事章节（核心问题 → 技术原理 → 方案全景 → 实现要点 → 失效边界），生成后立即执行深度自检
2. 自检不通过 → 当场修复 → 再次自检
3. 自检通过 → 输出简短结论（如"✅ 通过：去用/决策/诊断/洞察/映射/接引"） → 生成附录章节
4. 全部完成 → 全量复查

### 跨文档去重检查

全量复查时额外检查：
- 同一 Task 下的多篇文档是否有分析重复 → 保留在更相关的文档中，另一篇用"详见 xxx.md"引用
- 跨 Task 的文档是否有概念重复 → 在"前置知识"或"关联文档"中互相引用

## 设计思考

### 为什么锚点文档保留 5 章结构

当前 5 章（问题→原理→选型→实现→失效）是一条逻辑依赖链，对"从决策出发展开知识领域"这个使命是贴合的：
- 核心问题：锚定读者为什么需要关心这个领域
- 技术原理：展开领域知识的深度
- 方案全景：给出领域全貌（从"方案选型"改名而来，扩大了职责范围）
- 实现要点：从理论落地到实践
- 失效边界：指出知识的边界

这条线对技术深度展开型文档是自然的，不需要推翻。

### "方案全景"与"方案选型"的区别

旧版"方案选型"的隐性假设是"对比是为了证明项目选对了"，这导致：
1. 对比范围只覆盖项目考虑过的方案
2. 结论部分和 design.md 重复
3. 读者无法从中学到领域全貌

"方案全景"的定位是"给出该领域的方案全貌"，项目选择只是其中一个落地点。
这样读者获得的是通用知识，而非项目特供版。

### 行间接引 vs 知识地图的协作关系

行间接引服务阅读中的冲动——读者在某个点产生"想继续深入"的欲望时，即时响应。
知识地图服务阅读后的规划——读者需要全局视角，知道自己在知识网络的哪个节点。

两者做同一件事（接引），服务不同时刻。行间接引是散点，知识地图是收束。

### 为什么砍掉面试要点和 FAQ

面试考点的本质是知识的重点区，单独标注暗示"背重点"心态，和体系化搞懂的目标相悖。
FAQ 容易滑成概念定义，且其回答具体困惑的功能可以被行间接引和知识地图覆盖。

### 与 design.md 的边界：从"互补+引用"到"入口不同"

旧版边界策略的底层假设是"两者不能碰同一个决策"，导致技术文档的生存空间被压榨。
新版策略承认两者会触及同一个决策，但入口不同、讲法不同、深度不同：
- design.md 从"我们做了什么选择"出发，讲权衡结论
- 锚点文档从"这个领域是什么"出发，讲底层原理和体系化知识

重叠不再是问题，而是同一个决策从两个视角各讲各的。
