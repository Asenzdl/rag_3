# 阶段 1 设计日志规范

> 与 `project_info/task_execution_spec.md` 的阶段 1 流程配合使用。仅改动产出物和思考节奏，其余不变。

---

## 产出物

| 产出物 | 说明 |
|--------|------|
| `.project/tasks/phase_X/task_X.X_design.md` | 不变，最终架构决策 |
| `.project/tasks/phase_X/task_X.X_journal.md` | 新增，思考轨迹记录 |

journal 是 AI 在每个子步骤收敛后直接写入的文件。不是 thinking dump，不是草稿。只记录结构化的推理轨迹（约束筛选、方案对比、规则拉扯）。

---

## 执行流程

每个子步骤的节奏：

```
思考当前子步骤 → 收敛 → 追加 journal → 同步一行进度 → 继续下一步
```

同步信号不等待用户回复。用户看到后可介入，不说话则自动完成全部 8 步。

---

### 步骤 0：创建日志文件

进入阶段 1 时立即创建 `task_X.X_journal.md`。

### 步骤 1：理解需求

收敛后写入 journal：

```markdown
## 步骤 1：需求理解

任务概述：[一句话]
验收标准：[列表]
初步识别到的决策点：[名称] — 为什么像决策
```

对话输出：`步骤1完成，需求已理解，识别到 N 个决策候选`

### 步骤 2：构思初始方案

收敛后写入 journal：

```markdown
## 步骤 2：初始方案

模块划分：
- 新增：[文件]
- 修改：[文件]

核心数据流：[简述]

待深入分析的决策：[N 个，按优先级]
```

对话输出：`步骤2完成，初始方案已收敛`

### 步骤 3：需求对齐

使用 AskUserQuestion。用户确认后写入 journal：

```markdown
## 步骤 3：需求对齐

[问题] → [方案]
[问题] → [方案]
```

### 步骤 4：写出架构决策

每完成一条决策写入 journal：

```markdown
## 决策 N：[名称]

入口判定：[命中哪种]

候选方案：
- A：[优势 / 硬伤]
- B：[优势 / 硬伤]

分歧轨迹：
- 犹豫点：
- 拉扯的约束/原则：
- 天平倾斜的约束：
- 规则冲突：（无则写"无"）

收敛状态：未收敛 / 已收敛
```

无分歧则分歧轨迹写"无"。

对话输出：`决策N完成，倾向[X]` 或 `决策N有规则冲突，详见 journal`

### 步骤 5：反思检查

收敛后写入 journal：

```markdown
## 步骤 5：反思检查

面试知识点覆盖：✅/❌ [未覆盖项]
生产级注意事项覆盖：✅/❌ [未覆盖项]
10 维质量准则：[逐维标注通过/豁免+理由]
前瞻性设计报告：[预留点]
检查发现的卡点：[列表]
```

### 步骤 6：决策迭代

有修改时写入：

```markdown
## 决策迭代

修改：[名称]
原因：[反思发现的问题]
变化：[差异]
```

### 步骤 7：卡点对齐

使用 AskUserQuestion。写入对齐结果。

### 步骤 8：写出蓝图 + 其余章节

完成 design.md 后写入：

```markdown
## 结束

入口级决策：N 个
非关键决策：M 个
有分歧/冲突的决策：[列表]
```

对话输出：`阶段1完成，design.md 已输出`
