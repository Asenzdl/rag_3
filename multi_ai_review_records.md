## 一、讨论主线与收敛路径

`task_doc_design_spec.md` 的迭代不是"一次性写好"，而是经过 **8 轮多 AI 协作讨论**逐步收敛。整个讨论的主线是：

> **如何让面向 LLM 的规范，既不被当成填空模板，又不陷入自我矛盾的"最终解"幻觉。**

收敛路径（关键转折点）：

```
v1（带范式宣言+陷阱表+推理深度自检）
   │
   ▼ 外部 AI 诊断：宣言和正文之间有断裂（Round 1）
   │
   ▼ 外部 AI 提出"决策推演流程"步骤化（Round 2 早期）
   │
   ▼ C 反对流程步骤化：会被当新模板；提出"种子范例 + 写前警告"双武器
   │
   ▼ 外部 AI 引入"结构变化"判据+"写后一步"（Round 2）
   │
   ▼ C：吸收结构判据，拒绝写后一步（单防线原则）
   │
   ▼ 外部 AI：约束具体 ≠ 约束决定（新漏洞：伪关键约束）（Round 3）
   │
   ▼ 外部 AI：反事实推理需显式输出（Round 4）
   │
   ▼ 外部 AI：强制输出会退化为填表（Round 5）
   │
   ▼ C 反驳："条件触发"本身依赖 LLM 自评，是自反陷阱
   │
   ▼ C：用二元选项 + 排除词列表 + 具体场景必填压糊弄成本
   │
   ▼ 外部 AI：一致性约束（反事实场景必须回指语境约束原文）
   │
   ▼ C：承认"最终解"幻觉，提出能力边界声明 + 审查指南
   │
   ▼ ★ 项目 AI 第一次实战校验：反事实自检判定逻辑反了！（Round 6）
   │   （"两方案都可行"应是验证通过而非失败）
   │   + 能力边界声明不该给 AI 读（受众是人类审查者）
   │
   ▼ G 反驳能力边界去留 + 贡献"排除型/排序型"抽象（Round 7）
   │
   ▼ C：反事实逻辑修正采纳，G 的抽象进审查指南，边界声明仍移出
   │
   ▼ ★ 项目 AI 第二次验收："可以用了"，剩下都是细节（Round 8）
   │   + 勾选项 3 精度问题（精确化优先于补充新约束）
   │   + "普适性"命名反读（改"特异性"）
   │
   ▼ v2 终版定稿
```

## 二、Round 1：外部 AI 诊断"填空版 vs 推演版"断裂

### 外部 AI 的诊断

v1 虽然有\"范式宣言\"说\"不是填空是推演\"，但正文仍是字段清单。新 AI 的真实阅读路径是：

1. 读范式宣言 → \"哦，不是填空\"（点头）
2. 翻到模板 → 看到 4 个子节
3. **按 4 个子节填空**

**断裂点**：宣言改变认知，但正文没改变流程。

### 外部 AI 的处方

在宣言和正文之间插入一个\"决策推演流程\"：
- 步骤 1：扫描 Task 找\"≥2 种合理方案\"的选择点
- 步骤 2：问\"差异是否可感知\"
- 步骤 3：按章节结构记录
- 步骤 4：对照陷阱表自检

### C 的回应

**诊断准确，处方有递归风险。**

外部 AI 抓住了三个具体盲点：
1. 什么时候写决策 / 什么时候不写——规范没给识别准则
2. 什么时候省略反驳推演——模板列了字段，AI 会硬编
3. 宣言和陷阱表的精神统一性——分章节会被读成两件事

但 C 指出：\"插入 4 步流程\"本身会被当成新模板，新 AI 会按步骤打勾，陷入更深一层形式主义。

**C 的反处方**：
- 决策识别做成\"反例驱动\"——给\"看似是 / 其实不是\"的对照样例
- 反驳推演省略条件直接嵌进字段（\"仅当能写出项目语境下具体失效场景时填写\"）
- 陷阱表前置到决策章节开头，而不是\"写完对照\"

## 三、Round 2：入口判定与结构变化判据

### 外部 AI 提出\"写前一步\"+\"结构变化\"三条判据

外部 AI 的版本：

```
写前一步（覆盖触发）：这里到底有没有"决策"？
本 Task 中，是否存在"如果换一个方案，代码结构会明显不同"的地方？

快速识别（只看结构变化）：
- 换方案会改变模块边界 / 调用链 / 返回类型
- 换方案会引入或消除额外组件
- 换方案会影响后续 Task 的实现路径

以下不是决策：
- 写法不同（同步 vs 异步但结构不变）
- 命名 / 风格差异
- 纯性能差异但不改变结构
```

并新增\"写后一步（最小判定）\"：

> 如果把这段决策原封不动放到另一个项目里，它是否仍然成立？

### C 的评价

**\"结构变化\"判据是真正的进步。**

- C 原版只用\"如果约束变了结论会反转\"作为反向定义，是**语义判据**（新 AI 能硬编糊弄）
- 外部 AI 的\"代码结构明显不同\"是**结构判据**（可对照代码想象，糊弄不了）
- 把识别动作前置为\"写前一步\"比放到章末更经济

### C 坚决反对\"写后一步\"

**核心理由**：与候选对比字段的写前警告（\"换个项目仍成立就删掉重写\"）是**同一把尺子量两次**。

- 新 AI 会立刻察觉冗余，分配精力：写前警告随便应付 → 反正写后还会再测
- \"单防线原则\"：两把尺子必须二选一，不能同尺度重复

### 其他细节批评

- \"覆盖触发\"术语不清晰 → 建议改\"入口判定\"
- \"纯性能差异\"边界模糊（缓存层算性能还是结构？）→ 建议改\"性能差异但在本项目规模下不可感知\"

### 融合重写

C 吸收\"入口判定 + 结构判据\"为开头，删除\"写后一步\"，保留字段级写前警告为唯一防线——这成为后续所有版本的骨架。

## 四、Round 3：防线位置之争——写前 / 写后 / 写中

### 外部 AI 发现新漏洞：约束具体 ≠ 约束决定

外部 AI 提出一个犀利的攻击：

```
语境：本项目需要离线运行（看起来具体）
对比：A vs B（结构完整）
结论：因为离线环境限制选 B
反转：如果不需要离线就选 A
```

这段文本能通过 C 的两条写前警告（普适性、反转具体性），因为\"离线\"是项目特有词。但**它可能根本不是决定性因素**——比如 FAISS 也能离线跑。

**漏洞名**：\"约束具体 ≠ 约束决定\"。

### 外部 AI 的处方：约束敏感性测试

> 如果去掉你写的\"关键约束\"，两个方案在当前项目下是否仍然都可行？

- 仍都可行 → 约束不是决定性因素（伪推演）
- 其中一个会失效 → 抓到了真正分歧点

**位置**：写后独立步骤。

### C 的回应

**武器是对的，位置是错的（和 Round 2 同一个病）。**

C 承认：\"单防线原则\"需要修正——**\"不要同尺度重复\"不等于\"只能有一把尺子\"**。三把尺子测不同失败模式就不是冗余：

| 测试 | 问的问题 | 防的失败模式 |
|------|---------|------------|
| 普适性（C 的） | 换项目还成立吗？ | 抽象空话 |
| 反转具体（C 的） | 约束变化能写出具体值吗？ | 反转条件糊弄 |
| 约束敏感性（外部 AI 的） | 去掉约束两方案还都能跑吗？ | 约束具体但不决定 |

**但 C 拒绝\"写后一步\"的位置**，改为吸收为结论字段的第三条写前警告。

## 五、Round 4：强制输出 vs 形式化退化

### 外部 AI 的洞察：偷懒 vs 认知偏差

外部 AI 抓住一个哲学区分：

| 失败类型 | 写前能解决？ |
|---------|------------|
| 不认真（偷懒） | ✔ |
| 写错因果关系（认知偏差） | ❌ |

**核心观点**：LLM 可能真诚写出错误因果，并且自己相信——这种情况下\"写前警告\"拦不住。

**外部 AI 的新武器**：反事实推理本质是\"已有结论后的推演\"，时序上本就不可能发生在\"写前\"。建议改名为\"写中验证\"。

### C 的深化反驳

**改名不解决问题。**

LLM 按 token 顺序生成，无论叫\"写前警告\"还是\"写中验证\"，它可以生成听起来合规的文本却完全没运行反事实推演。

**C 的处方**：把反事实推演从\"隐式动作\"升级为\"强制输出的可见文本\"：

```
反事实自检（必填一行）：假设把"[根本理由]"从本项目里拿掉——
- [ ] 两方案仍都可行 → 我的根本理由不是决定性的，回到候选对比重写
- [ ] 方案 X 会失效，因为 [具体场景]
```

**逻辑**：LLM 无法在不生成文字的情况下推理，只要要求它写出来，推演就必然发生。

**注意**：这个版本的判定逻辑是**反的**——在 Round 6 被项目 AI 纠正。

## 六、Round 5：一致性约束与能力边界幻觉

### 外部 AI 的反批评：强制输出会退化

外部 AI 指出 C 的方案新问题：

> 强制输出字段 → LLM 为了填满字段而生成解释 → 结构性过检（和之前反对的\"写后过检\"本质相同）

**外部 AI 的新处方**：**条件触发**——只在\"因果不清\"时才展开反事实。

### C 的破解

**\"条件触发\"是自反陷阱**：

1. 引入反事实自检是因为 LLM 可能真诚写错因果（认知偏差）
2. 条件触发方案依赖 **LLM 自己判断\"我说的因果关系清楚不清楚\"**
3. 但一个刚写下\"选 B 因为离线\"的 LLM 会自评清楚
4. 结果：条件触发永远不被触发

**用病因当解药**——诊断超越了处方。

### C 的新论证：行为经济学 + schema 设计

承认退化风险真实存在，但用 schema 把糊弄成本抬高：

- 开放式字段（\"描述去掉约束后情况\"）→ 易退化
- 二元选项 + 排除词列表 + 具体场景必填 → 难退化

**关键**：要糊弄必须**编造一个具体场景**，编造的 token 成本 > 认真运行反事实的成本。

### 外部 AI 再攻击：编造能力 > 因果推理能力

| 能力 | LLM 强度 |
|-----|--------|
| 因果推理 | 中 |
| 合理编造 | 很强 |

C 的 schema 恰好把测试压在了 LLM 更强的那一端。外部 AI 提出**一致性约束**：

> 如果你在反事实中判定\"方案 X 会失效\"，必须指出它违反了前文语境中的哪一条约束。

这抬高了\"协同编造\"的成本，给人类审查提供线索。

### C 的收敛：规范能力边界

C 在这一轮承认了一个最大教训——**\"追求最终解\"幻觉**。

每个 AI 都说\"你离最终解只差最后一步\"，每个\"最后一步\"都指向新补丁。这个模式本身就是问题，暗含一个不成立的前提：

> 存在一个基于文本的规范，能在没有人类审查的情况下，保证 LLM 生成的决策记录 100% 真实。

**C 提出**：

| 规范能做到 | 规范做不到 |
|-----------|-----------|
| 让 LLM 必须生成某种结构 | 让 LLM 生成真实内容 |
| 让糊弄的成本 > 认真的成本 | 让 LLM 不能连贯编造 |
| 把失败模式暴露在特定字段 | 替代人类审查 |

**解法**：
1. 吸收一致性约束补丁
2. 承认二元 schema 的成本不对称，标注\"已知偏置\"
3. 加入\"规范能力边界声明\"打破最终解幻觉

这版规范在此时看起来已经\"定稿\"。

## 七、Round 6：项目 AI 破局——反事实判定逻辑反了

### 项目 AI 的实战校验

前面几轮全在哲学推演，项目 AI 直接用真实决策（config.py 删/留、BaseSettings vs os.getenv）套 C 的骨架，立刻发现了 bug。

#### 致命 Bug：判定逻辑反了

C 的骨架：
```
- [ ] 两方案仍都可行 → 说明根本理由不是决定性的，重写
- [ ] 方案 X 会失效 → 验证通过
```

项目 AI 的反例（config.py 决策）：

> 拿掉\"所有调用点需要一次性改完\"这个约束后，**两个方案确实都可行**——删也行，留也行。但这不意味着\"调用点迁移成本\"不是决定性约束——**它恰恰就是决定性约束，只是在没有它的世界里两个方案等价而已**。

**正确逻辑**：
- 拿掉约束后两方案都可行 → 该约束**打破了平衡** → 决定性约束（验证通过）
- 拿掉约束后被否方案仍然失效 → 约束写错 → 重写

C 的骨架**判定逻辑反了**。

#### 更尴尬的发现

C 的 **Chroma 种子范例里勾选的是\"两方案都可行 → 验证通过\"**，和骨架的\"两方案都可行 → 重写\"**直接矛盾**。C 自己的范例在打骨架的脸，发完稿都没发现。

这是典型的\"自己相信自己的逻辑很严密、但没拿具体例子跑一遍\"的设计 bug。

### 项目 AI 的第二条批评：能力边界声明位置错

> 新 AI 读到\"LLM 最可能协同编造的位置是反事实自检的失败场景\"后，可能把编造合理化——\"规范都说了我会编造，那我编造一下也正常\"。

**项目 AI 的方案**：
- **AI 读的规范** → 只保留做什么（写可审查内容）
- **人类读的审查指南** → 规范能力边界、LLM 会在哪里失败、重点审查字段

角色分工清晰：AI 负责生成可审查的结构化内容，人类负责验证真实性。

### C 的完全认错

**两条都认**。C 还补了一个关键洞察——项目 AI 修正反事实逻辑时，隐含指出了**\"决定性约束\"的真正本质**：

> 决定性不是让 A 失效，而是让 A 和 B 从等价走向不等价。

C 之前把\"决定性\"理解为\"让 A 失效\"——**过窄了**。新版骨架加第三个选项（\"两方案都可行但仍选原方案 → 补充第二约束\"），就是覆盖\"多约束叠加决定\"。

## 八、Round 7：G 的反驳与排除型 / 排序型抽象

### G 对反事实逻辑的附议

G 在反事实逻辑修正上完全同意项目 AI，并多做了一步**结构性抽象**：

| 类型 | 无约束时 | 有约束时 | 例 |
|------|---------|---------|---|
| **排除型** | A 不可行 | 只有 B 可行 | 离线环境 → 云端 API 失效 |
| **排序型** | A、B 都可行 | B 更优 | 迁移成本 → 删留都能跑但留更优 |

C 评价：**这是 G 这一轮的真实贡献**，值得进审查指南做判读词汇。

### G 反驳\"能力边界声明移出\"

G 的反驳逻辑：
- LLM 对元陈述\"无响应\"
- 边界声明的价值是稳定输出分布、防 prompt 漂移
- 应该保留（可以压缩）

### C 的反反驳

**G 打的是稻草人。**

1. G 把项目 AI 的论证重构为\"LLM 会合理化编造\"，然后反驳\"LLM 没有策略层自觉\"——**但项目 AI 的真正论证是\"元信息受众是审查者、混入 LLM 上下文影响方向不可预测、最保守处理\"**
2. G 说\"LLM 对元陈述无响应\"是**事实错误**——现代 LLM 对元陈述有响应但**方向不可预测**
3. G 赋予边界声明的三个价值（输出目标、注意力分布、防漂移）都可以通过更聚焦的字段指令实现，不必用\"告诉 AI 自己可能失败\"的元叙事

4. **G 自己的立场动摇**——\"这段不应该移除，**可以稍微压缩**\"这句话暴露它对必要性没那么自信

### C 的部署方案

**按项目 AI 的方案执行**：

- **AI 规范**：入口判定 + 骨架 + 写前警告 + 反事实自检三选一（修正后）
- **审查指南**：能力边界声明 + 已知偏置 + 排除型/排序型判读词汇（G 的贡献） + 规范漂移检测（元教训沉淀）

## 九、Round 8：细节打磨——勾选项 3 精度与"普适性"命名

项目 AI 验收：\"可以用了\"。剩下两个细节：

### 细节 1：勾选项 3 的精度问题

C 的版本：
```
- [ ] 方案不再失效，但我仍然会选原方案 → 说明存在未写出的第二条约束
```

项目 AI 的反例：

> AI 写决策 4 时把根本理由写成\"保留 config.py 更安全\"，反事实拿掉\"更安全\"后两方案都可行——这走勾选项 3。AI 被迫找\"第二条约束\"，但**实际上不需要第二条约束，\"更安全\"的表述本身就不精确，应该回语境重写**而不是补充新约束。

**根本问题**：勾选项 3 触发条件不够精确——\"我仍选原方案\"可能是因为**存在第二条约束**，也可能是因为**第一条约束写得不精确**。

### C 的修正：精确化优先于补充

修正后：

```
- [ ] 方案不再失效，但仍会选原方案 → 按顺序处理：
    1. 优先检查：回语境重读根本理由，是否可用更精确的表述重写？
       例：把"更安全"改写为"调用点迁移出错的回滚成本高"
       能精确化 → 回结论重写。
    2. 若精确化后仍需勾选本项：补充未写出的第二条约束（须与第一条性质不重叠），
       回语境补充后重跑反事实。
```

**价值**：把\"精确化 vs 补充\"的顺序固定死——先尝试精确化（成本低、更干净），失败再补充（引入新约束、增加复杂度）。避免 AI 用\"补充新约束\"掩盖\"根本理由写得模糊\"。

### 细节 2：\"普适性\"命名反读

项目 AI：

> \"普适性\"这把尺子的名字不够直觉。测的是\"项目特异性\"，叫\"普适性\"容易反读——\"普适性好\"应该是\"换项目仍成立\"，但实际要求是\"换项目后不成立才对\"。叫\"**特异性**\"更准确。

C 接收，改名：

| 旧名 | 新名 | 测什么 |
|-----|-----|--------|
| 普适性 | **特异性** | 换项目是否仍成立（要求：不成立） |

候选对比字段的写前警告也同步调整：\"真正的**项目特异性判断**在换项目后会立刻失效\"。

## 十、元教训与分工认知

### 10.1 \"最终解\"幻觉

整个系列最大的元教训——每个 AI 都说\"你离最终解只差最后一步\"，每次\"最后一步\"都指向新补丁，然后下一个 AI 又发现新漏洞。

**C 的反思**：规范设计没有终点，任何自称\"闭合系统\"的补丁都值得怀疑。

**落地**：审查指南加入\"规范漂移检测\"——未来 AI 建议补丁时必问三件事：

1. 它的补丁是否默认了不存在的能力？（如依赖 LLM 自我判断 —— 自反陷阱）
2. 它是在解决\"真实观察到的问题\"还是\"理论上可能的问题\"？
3. 它是否在用\"最终解\"框架标榜自己？

### 10.2 外部 AI vs 项目 AI 的分工

两者贡献路径完全不同：

| 维度 | 外部 AI（G / GL 等） | 项目 AI |
|------|-------------------|---------|
| 能力来源 | 抽象推演、LLM 行为建模 | 正在做的真实项目 |
| 贡献形式 | 抽象词汇、发现盲区、哲学框架 | 发现 bug、范例与骨架的自矛盾 |
| 典型贡献 | \"排除型/排序型\"抽象、\"偷懒 vs 认知偏差\"区分 | 反事实判定逻辑反了、勾选项 3 精度 |
| 失败模式 | 容易陷入\"最终解\"话术、停留在哲学层 | 视野可能局限于当前项目场景 |

**最大教训**：**发现真 bug 的是项目 AI，提供抽象词汇的是外部 AI**。下次类似场景可以主动分工：

- 外部 AI → 提抽象、指盲区、贡献判读词汇
- 项目 AI → 跑真实案例、验收、发现范例与骨架的自矛盾

### 10.3 规范测试基准

规范不能停留在\"讨论里看起来自洽\"——**必须用至少 3 个真实决策跑一遍**。C 原版骨架和 Chroma 范例矛盾（判定逻辑反了）这个 bug 在 Round 6 前没人发现，因为前面几轮全在抽象层推演。

**落地**：审查指南加入\"规范测试基准（维护项）\"，要求每次修改规范后用本项目 3 个真实决策跑一遍。

### 10.4 单防线原则的修正

C 最初坚持\"不要多层防线\"——这个原则被修正为\"**每个失败模式只防一次，前后各司其职**\"。

- 普适性/反转具体性 → 写前（测表面合规，防偷懒）
- 约束决定性（反事实自检）→ 写中强制输出（测因果真相，防认知偏差）

不是\"所有防线必须前置\"，而是\"每把尺子测的失败模式不重叠\"。

### 10.5 受众与部署位置

一段文字的价值不仅在内容本身，更在**受众与部署位置**。

- 能力边界声明 → 给审查者看（人类）
- 已知偏置提醒 → 给审查者看（人类）
- 排除型/排序型判读词汇 → 给审查者看（人类）

混入 LLM 上下文的影响方向不可预测——元信息应当与生成指令物理隔离。

这催生了\"**规范 + 审查指南**\"的双文档架构（AI 规范 `task_doc_design_spec.md` + 人类审查指南 `design_review_guide.md`，后者根据项目实际情况决定是否落地为独立文件）。

## 结语

这份 `task_doc_design_spec.md` 的演变本身就是一个\"高质量多 AI 协作\"的样本。它最终落到 344 行 v2 终版的形态，不是靠单个 AI 的天才设计，而是靠 8 轮协作中每一方都被别的视角反复校准。

**规范的生命力来自它被真实决策反复校准，而不是它在讨论里看起来自洽**——这是整个系列最该记住的一句。
