## 二、v1 → v2 初版：从知识指引到推演引擎

### 触发契机

根据 LLM 遵循度与文档长度的分段分析（<150 行高遵循、150-250 行需锚点、250-350 行风险、>350 行疲劳），v1 的 265 行已接近临界。真正的问题不是"太长"，而是：**v1 中有大量对人类有用、但 LLM 会忽略的元叙事**，而关键约束却藏在段落里没被锁死。

### 核心策略

区分"解释性元叙事"（必砍）与"行为边界约束"（必留且压缩至开头 3 行硬约束）。

### 砍掉的部分（元叙事外置）

| v1 章节 | 处置 | 理由 |
|---------|------|------|
| **0、范式宣言** | 整章删除 | "不是填空题而是推演引擎"属于元叙事，应用行为约束替代宣言 |
| **AI 写决策分析最易踩的 4 类陷阱**（表面合理 / 伪选型 / 决策前置 / 方案膨胀） | 整表删除 | 分类式陷阱列表对 LLM 是"看过就忘"，不如字段级写前警告 |
| **B. 推理深度自检（可判定追问）** | 整节删除 | 三条追问（决策考古 / 反驳痕迹 / 陷阱对照）是人类自查工具，LLM 会形式化应付 |

### 新增的部分（行为锁定）

| 新增模块 | 位置 | 作用 |
|---------|------|------|
| **硬约束 4 条** | 顶部（先读） | 代替"范式宣言"，直接列不可违反的底线 |
| **入口判定（动笔前必答）** | 决策章节开头 | 用三条操作性判据（模块边界 / 组件增删 / 影响后续 Task）+ 四条"不是决策"排除项过滤，全不命中则整章不写，防止硬凑决策稀释密度 |
| **种子范例：推演版 vs 填空版** | 决策章节 | 用 Chroma vs FAISS 的完整正例 + 反例对比，展示"扎根本项目约束"vs"任何项目都成立"的区别 |
| **字段级写前警告**（斜体） | 每个决策字段下 | 语境 / 候选对比 / 反驳推演 / 结论各配一条写前警告，动笔前先读，命中即回炉。结论字段的警告合并为一条：空泛表述回语境重找 + 反转条件必须精确到"哪条约束变成什么值" |
| **反事实自检三选一**（强制） | 决策结论后 | ① 约束拿掉后方案不再失效 → 验证通过；② 仍失效 → 失效原因不是你写的约束，回炉；③ 不再失效但仍选原方案 → 优先局部精确化，若失败则补充未写出的第二条约束 |
| **代码蓝图"为什么"四类触发条件** | 代码蓝图开头 | 设计决策 / 反直觉辩护 / 功能取舍 / 替代方案排除，配写前警告（换项目仍成立 → 删除） |
| **数据变换例外** | 赋值和运算规则下 | "复杂数据变换可直接写出关键表达式"——解决"注释精度要求冲突"（需精确到可翻译 vs 禁写代码实现）问题 |
| **高密度种子范例（DocumentProcessor，30 行）** | 代码蓝图章节末 | 在一个完整函数中同时覆盖 docstring 双类为什么、三类步骤（流程控制 / 数据变换 / 业务校验）、条件分支树、质量标注四维、数据变换例外，通过对比自然呈现精度差异 |

### 结构性决策要点

1. **质量标注就近分布**：鲁棒性 / 可观测性 / 可测试性 / 可扩展性四维必须放在步骤层注释，不上移到函数/类层——因为物理位置与异常分支、日志、Mock 注入、TODO 标记强绑定，上移会导致描述抽象化。
2. **例外条款优先于分类系统**：面对"注释精度冲突"，不引入 D 提出的"三档精度分类"或 G 提出的"函数级上移"，而是用"精准例外 + 多样化种子范例"双轨。

### 量化结果

- 行数：265 → 334（+69）
- 但**约束密度显著提升**：硬约束从分散论述压缩为 4 行清单；决策字段从 4 个变 5 个（每个都带写前警告）；种子范例从 0 个变 2 个（推演/填空对比 + DocumentProcessor 完整示例）。

## 三、v2 初版 → v2 终版：边界分工与学习者适配

### 触发契机

第一版 v2 出炉后，进一步审查发现：
- 错误处理策略章节的"为什么保留"说明是元叙事，应改为"与代码蓝图的分工"；
- 用户画像是学习者而非成熟架构师，docstring 缺一个承载"非标准类比、面试视角"的兜底字段；
- 条件分支规则只说"路径 >2 条用分支树"，没说 ≤2 条怎么办；
- 质量自检缺两条关键客观项。

### 改动清单

| # | 位置 | 改动 | 动机 |
|---|------|------|------|
| 1 | 文档头 | 新增"**范例说明**：下方种子范例中出现的 `Task 4.2`、`DocumentProcessor`、`embedder` 等均为教学构造" | 防止 LLM 把教学范例中的符号当成真实约束 |
| 2 | 错误处理策略 | 新增"**与代码蓝图的分工**：本章节给出异常的全局策略表；代码蓝图的步骤注释只内联到实例级描述" | 从"为什么保留"元叙事改为边界分工定义 |
| 3 | 交互时序图 | 新增"**与代码蓝图的分工**：时序图捕捉跨组件协作流；条件分支树捕捉单个函数内部逻辑流" | 同上 |
| 4 | docstring 字段 | 新增"**其他**（可选）：上述字段无法承载、但对读者理解有帮助的内容（如非标准类比、面试视角延伸）。*写前警告：如果换一个项目/函数仍然成立，删除。*" | 学习者画像适配；警告与架构决策的置换测试同构，不增理解成本 |
| 5 | 格式规则 #2 | 拆分为：路径 ≤ 2 条直接写 if/else；路径 > 2 条使用条件分支树 | v2 初版只说"> 2 条时用分支树"，遗漏 ≤ 2 条场景的明确指引 |
| 6 | 质量自检 | 新增两条：异常步骤标注处理方向（回退/传播/包装）；副作用/关键路径步骤标注日志（级别+字段） | 补齐客观勾选项的覆盖盲区 |

### 量化结果

- 行数：334 → 344（+10）
- 改动性质：全部为"消歧 + 归位"级别的精修，无结构性重构。

## 六、设计哲学总结

| 原则 | 体现 |
|------|------|
| **元叙事外置** | 删"范式宣言"、删"AI 陷阱表"、删"推理深度自检"；"为什么保留"说明改写为"与代码蓝图的分工" |
| **行为锁定** | 硬约束顶置 4 条；字段级写前警告替代段落级劝诫；反事实自检三选一强制勾选 |
| **高密度范例** | 推演/填空对比范例（决策章）+ 30 行 DocumentProcessor（蓝图章）+ 范例读法注解 |
| **精准例外替代分类** | 数据变换例外 + 模板/常量例外 + 路径 ≤2/>2 分档；不引入三档精度分类 |
| **就近分布** | 质量标注四维强制放步骤层，不上移 |
| **学习者画像适配** | docstring 新增"其他"兜底字段，配置换测试警告 |
| **边界分工显式化** | 错误处理策略与蓝图步骤注释的分工、时序图与条件分支树的分工 |

## 七、多 AI 讨论轨迹（简要）

这份规范从 v1 到 v2 终版的演变，本质是一场为期数小时的多 AI 协同评审：由**项目 AI（C）**主笔、**外部 AI（G / GL）**挑刺、**用户**把关方向，经 8 轮来回收敛成稿。完整原文与逐轮洞察见：👉 [`multi_ai_review_records.md`](./multi_ai_review_records.md)

### 8 轮讨论转折点

| 轮次 | 主轴 | 关键转折 |
|------|------|----------|
| **R1** | 填空 vs 推演断裂诊断 | 外部 AI 指出 v1 的"4 字段骨架"+"自检问题"本质是填空题，无法迫使真实推演 |
| **R2** | 入口判定成型 | 从模糊的"重要决策才写"落地为三判据 + 四排除项的可执行判定 |
| **R3** | 防线位置 | 争论"字段级写前警告" vs "独立推理追问"，最终选字段级就近嵌入 |
| **R4** | 强制输出的形式化风险 | 识破"反事实自检若写成填空会退化"，引入三选一勾选 + 精确到约束名值 |
| **R5** | 能力边界幻觉 | 外部 AI 提出的"一致性约束"被识别为超出 LLM 因果推理能力 |
| **R6** | ⭐ **项目 AI 破局** | 发现 C 自己写的反事实判定逻辑反了——Chroma 种子范例与骨架判据自相矛盾，修正后逻辑自洽 |
| **R7** | 排除型 vs 排序型抽象 | G 反驳精度诉求，C 接受将约束分类为"排除型/排序型"提升抽象层 |
| **R8** | 细节打磨 | 勾选项 3 精度校准、"特异性" → "普适性"命名、docstring 兜底字段 |

### 元教训（摘录）

- **没有"最终解"幻觉**：每轮被肯定为"自洽"的版本，下一轮仍会被挑出漏洞；规范演化是持续收敛，而非一次性封版。
- **AI 分工**：外部 AI 擅长"挑刺找漏洞"（破），项目 AI 擅长"扎根项目约束整合"（立），两者不可互替。
- **单防线原则修正**：不是"只设一道防线"，而是"每类风险只设一道且放在最近位置"——字段级警告是行为级防线，反事实自检是结论级防线，两者在不同层级各司其职。
- **受众与部署位置耦合**：规范（面向 LLM 执行）与审查指南（面向人类复核）必须物理分离，合并会稀释密度。

> 完整 Round 1-8 原文、各方立场、每轮关键洞察，详见 [`multi_ai_review_records.md`](./multi_ai_review_records.md)。
