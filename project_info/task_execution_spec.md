# Task 执行流程规范

> 触发条件：当用户说 **"开始 Task X.X"** 时，读取本文件并按流程执行

---

## 总览：三阶段 + 审核模式

```
阶段1 反思驱动架构设计 → 阶段2 完整代码实现 → 阶段3 技术学习文档
```

审核模式控制阶段间的暂停节奏（用户在触发时指定）：

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| **模式 A**（默认） | 阶段1+2 连续执行 → 暂停确认 → 阶段3 分轮生成 | 平衡效率与质量 |
| **模式 B** | 每个阶段完成后暂停确认 | 想逐阶段审查 |
| **模式 C** | 阶段1+2 连续执行 → 暂停确认 → 阶段3 分轮生成 + 完成后调用 code-reviewer 审查 | 最高质量保障 |

触发示例：`开始 Task 1.5` → 模式A | `开始 Task 1.5 模式B` → 模式B | `开始 Task 1.5 模式C` → 模式C

---

## 阶段 1：反思驱动架构设计

### 输入

- 当前 Task 的 outline 文件（`.project/outline/phase_X/task_X.X_*.md`）
- 应该按照 `project_info/task_doc_design_spec.md` 中的 **design.md 模板** 编写，但可以更好的发挥
- 按 `CONTEXT_INDEX.md` 中的 Task 目标判断需读取的 Task 文档（按需加载，用于前瞻性设计和指出过往设计的不足）

### 过程（AI 内部反思，严禁生成代码文件）

1. **理解需求**：读取原始 task 需求，理解任务目标、面试知识点、生产级注意事项、验收标准
2. **构思初始方案**：设计模块结构、核心接口、数据流
3. **反思检查**（按顺序，每步不通过则迭代优化方案）：
   - **面试知识点全覆盖？** — 检查 task 中列出的每个面试知识点，确认架构中有对应的设计体现
   - **生产级注意事项全覆盖？** — 检查 task 中的每条注意事项（异常处理、配置管理、性能等），确认架构中有对应方案
   - **10维最佳实践充分落地？** — 根据 CLAUDE.md「质量准则：MUST」中逐维检查，确认每个维度在本 Task 中有具体、可执行的落地方式或**给出豁免声明**
4. **整理最终架构设计**：MUST 架构决策覆盖度高且有理有据，MUST 代码蓝图达到"施工图纸"精度

### 产出

`.project/tasks/phase_X/task_X.X_design.md`

- **严禁包含**：完整实现代码、测试代码（这些在阶段2直接生成文件）
- 此文档的核心目标：深刻理解所有核心架构决策，根据代码蓝图独立编写代码

---

## 阶段 2：完整代码实现

> 对同一文件的多个编辑应合并为尽量少的操作。

- 直接生成/修改 `src/` 下的代码文件
- 直接生成 `tests/` 下的测试文件，但不用替我测试
  - 覆盖正常路径、边界情况、异常路径
- 更新必要的 `__init__.py`（导出公共接口以及提供注释）
- **禁止将代码输出到对话框或写入文档**

---

## 阶段 3：技术学习文档

- 生成 `.project/docs/task_X.X/` 下的技术文档（按当前 Task 编号创建子文件夹）
- 应该按照 `project_info/tech_doc_design_spec.md` 中的 **技术文档模板 + 深度标准** 编写，但可以更好的发挥
- 面向用户学习，由主 AI 生成，质量要求：
  - 技术原理要**深入本质**，不要浅尝辄止
  - 方案对比要有**明确的选择指南**
  - 代码示例必须**完整可理解**
  - 最佳实践要有**推荐做法 + 反模式**对比
  - **渐进式建构**：复杂技术需展示从简到繁的演进过程（如 V0→V1→V2），而非直接给出最终结果
  - **失败模式分析**：必须覆盖该技术的典型失败场景、边界情况和诊断方法
  - **理论解释**：不能停留在"怎么用"，必须解释"为什么有效"（底层机制/数学直觉）

### 分轮生成机制

1. **规划轮**：列出本 Task 需要生成的所有技术文档清单（文件名 + 主题 + 预计深度要点）
2. **逐篇生成**：每轮只生成**一篇**技术文档，生成后立即执行深度自检（按 `project_info/tech_doc_design_spec.md` 中的自检问题）
3. **深度自检**：去用测试、决策测试、诊断测试、洞察测试；通过后再生成下一篇；自检不通过则当场修复
