# Task 执行流程规范

> 触发条件：当用户说 **"开始 Task X.X"** 时，读取本文件并按流程执行

---

## 总览：三阶段 + 审核模式

```
阶段1 反思驱动架构设计 → 阶段2 完整代码实现 → 阶段3 技术学习文档
```

审核模式控制阶段间的暂停节奏（用户在触发时指定）：

| 模式 | 行为 | 
|------|------|
| **模式 A**（默认） | 阶段1+2 连续执行 → 暂停确认 → 阶段3 分轮生成 | 
| **模式 B** | 每个阶段完成后暂停确认 | 
| **模式 C** | 阶段1+2 连续执行 → 暂停确认 → 阶段3 分轮生成 + 完成后调用 code-reviewer 审查 | 

触发示例：`开始 Task 1.5` → 模式A | `开始 Task 1.5 模式B` → 模式B |

## 阶段 0：理解用户画像

- **本质**：将用户画像作为全阶段所有理解和约束的解读校准器
- **必须重述你对用户画像的理解**

## 阶段 1：理解新增需求

**目标**：纯粹理解需求，不提前收敛方案。

请阅读当前 Task 文件（如 `.project/outline/phase_X_*/task_X.Y_*.md`），完成以下步骤：

1. **提取任务目标**：系统需要新增什么能力？新增了什么复杂度？
2. **理解面试知识点**
3. **区分目标与方案**：哪些描述是功能目标（必须实现），哪些只是候选实现（可以替换）。
4. **识别新增复杂度来源**，重点关注以下维度的变化：
   - 状态管理、生命周期、并发、checkpoint、memory
   - 外部副作用、streaming、interrupt、retry、cache 等
5. **查询最佳实践**：必须查阅相关官方文档与社区公认实践，但不得因官方示例存在就默认采用。

**本阶段禁止**：
- 提前进入方案收敛或 tradeoff 讨论。
- 默认接受 Task 中的具体实现。

---

## 阶段 2：Discovery（搜索空间展开）

- 用户需求中的**功能目标**默认可信，直接接受。
- 用户需求中的**具体实现方案**仅是候选路径，必须质疑、对比，不得直接采纳。
- 现有代码结构不具备天然正确性，必须重新评估其是否仍适合当前任务。

### 根据下表**枚举**所有待决策点

| 维度 | 决策点 |
|------|--------------|
| 边界 | 功能放哪个文件/模块？单文件还是拆分多文件？几种方案各有何取舍？ |
| 分层 | 新功能属于哪个层？依赖方向应该怎样？ |
| 依赖来源 | 依赖有几种获取方式（闭包/参数/工厂/模块级导入）？分别适用于什么测试场景？ |
| 调用顺序 | 节点间有无数据依赖？无依赖的操作是否可以调整顺序或并行？有几种拓扑方案？ |
| 接口契约 | 函数/节点的入参和返回值长什么样？最小必要集是什么？有几种合理设计？ |
| Schema | 是否需要新增状态字段？类型和默认值有几种选择？谁负责维护这个状态的一致性（所有权归属）？ |
| 生命周期 | 资源（连接/模型/DB会话）谁创建、谁持有、谁释放？几种归属方式的利弊？ |
| 异常处理 | 有几种失败方式？每种失败的回退方向有几个合理选择？错误是就地消化还是传播给调用方？传播范围？是否要重试？重试策略？ |
| 配置化 | 新引入哪些可变参数？哪些进 Settings、哪些硬编码？默认值选什么？ |
| 框架API | LangChain/LangGraph 是否提供内置函数做这件事？有几种选择——直接用 / 包装 / 自实现？ |
| 扩展预留 | 当前选择会影响后续哪个 Task 的路径？影响程度多大？ |

---

### 待决策点记录

对 11 个维度逐个产出条目，每维一条。三种状态：

- **不适用** + 具体理由：该维度在当前 Task 中不存在决策空间
- **无真实分歧** + 简要说明：存在选择但无结构影响，不值得进入 Filtering
- **候选项** + 候选方向 + 影响范围：存在多个合理方向，进入 Filtering 判断


根据以下格式写入 `.project/task_X.X/task_X.X_scan.md`

```markdown
# 待决策点记录
## {维度名称}

状态：{不适用 / 无真实分歧 / 候选项 D{编号}}

{不适用 → 具体理由}
{无真实分歧 → 简要说明为什么有选择但无关紧要}
{候选项 → A: xxx / B: xxx / C: xxx}
{候选项 → 影响范围：具体文件或模块}
```

### 扫描完成条件

进入 Filtering 的前提（缺一不可）：
- [ ] 11 个维度均已产出条目（无遗漏）
- [ ] 每个条目有明确状态标记
- [ ] 状态为"不适用"的条目有具体理由
- [ ] 状态为"候选项"的条目有至少 2 个候选方向 + 影响范围
- [ ] 状态为"候选项"的条目尚未出现 tradeoff 判断或偏好声明

## 阶段 3：Filtering（分歧过滤）

**目标**：对 scan.md 中标记为"候选项"的 Dn，逐项判定是否构成真实架构分歧。

### 三源检查
逐项检查每个 Dn 的三个源：
- 水平源：需求内部是否存在多个合理方向？
- 质量源：候选方案是否与 10 维存在张力？
- 上下文源：候选方案是否改变模块边界/调用链/返回类型/生命周期/后续 Task？

### 产出
每个 Dn 标记一个结果：
- 无分歧 → 不入后续分析
- 分歧 → 进入深挖

### 强制要求
- 所有 Dn 完成三源检查后，才能进入下一阶段
- 不可因发现某个 Dn 分歧而跳过剩余检查

→ 写入 `.project/tasks/phase_X/task_X.X_filter.md`
→ 用 [AskUserQuestion] 展示全部过滤结果，用户确认后进入深挖，深挖单独拆为阶段 4：

## 阶段 4：Deep Analysis（架构深挖）

**目标**：对 filter.md 中标记为"分歧"的决策点，逐个做止损器 5 问 + 候选对比。

**前置条件**：filter.md 已确认

### 止损器 5 问
1. 各方方案成立依赖什么前提？
2. 这些前提在当前项目中是否成立？
3. 生命周期和 ownership 是否匹配？
4. 错误会如何传播？
5. 当前判断在什么条件下需要重审？

### 候选对比
每个分歧点输出：
- 候选方案（A/B）
- 本项目优势/硬伤
- 反驳推演
- 结论 + 反转条件

→ 写入设计文档对应章节
→ 用 [AskUserQuestion] 确认每个分歧的最终决策


## 阶段 3：Filtering（分歧过滤）

逐个检查待决策清单中的每个决策点是否会导致决策分歧？
- 检查水平源（需求内部）：Task 所有具体方案是否存在多个合理方向？
- 检查质量源（第 2 层修正权）：激活约束优先级的"第 2 层可修正第 3 层" → Task 所有具体方案是否与 10 维质量准则存在张力？
- 检查上下文源（现有架构）：Task 所有具体方案是否与现有架构存在张力？

[强制要求]
- [三源均无问题] → 必须用 [AskUserQuestion] 确认无分歧后进入阶段 3
- [任一有问题] → 必须启动[架构拉扯止损器] 5 问 → 必须用 [AskUserQuestion] 呈现分歧点；必须确认无分歧后进入阶段 3

## 阶段 4：架构设计

### 过程（严禁生成 `src/`、`tests/` 下的代码文件）

> 核心认知：必须思考用户画像，理解用户需求

1. **构思初始方案**：设计模块结构、核心接口、数据流

2. **写出架构决策**：按 `task_doc_design_spec.md` 的"架构决策与权衡"章节格式，写出决策章节（仅决策，不含蓝图）。写出来才能被检查和推翻，而不是在思维中隐式漂移

3. **反思检查**（对照已写出的决策，逐项通过或标记为卡点）：
   - **10维最佳实践充分落地？** — 按 CLAUDE.md「质量准则」逐维检查。先判断是否适用，不适用则豁免；适用再检查落地方式
   - **前瞻性设计报告** — 报告验收标准外的潜在预留点选项，在卡点对齐中呈现
   - **非关键决策遗漏检查？** — 过一遍"本 Task 中做出的、不是入口级但可能让后续读者困惑的选择"
   - **卡点标记**：遇到架构决策且 3 轮内无法自行收敛时，创建待确认标记，继续后续检查

4. **决策迭代**：反思检查发现问题的 → 修改决策 → 重跑检查；无修改 → 决策稳定，进入下一步

5. **卡点对齐**：
   - 如果无卡点，跳过此步
   - 使用 AskUserQuestion 将每个卡点（含前瞻性设计报告中的预留点）作为选择题呈现
   - 用户确认后修正方案，卡点全部解决后进入下一步

6. **写出蓝图 + 其余章节**：基于稳定决策，写完剩余章节
   - 蓝图暴露决策不自洽之处时，回步骤 4 修改决策

### 代码蓝图粒度

按 `project_info/task_doc_design_spec.md` 中"代码蓝图"章节执行

### 输出

- **核心目标**：深刻理解所有核心架构决策，根据代码蓝图独立编写代码
- 存储于 `.project/tasks/phase_X/task_X.X_design.md` (目录已存在，直接生成新文件)

## 阶段 5：完整代码实现

> 对同一文件的多个编辑应合并为尽量少的操作。

- 直接生成/修改 `src/` 下的代码文件
- 直接生成 `tests/` 下的测试文件，必须在阶段3技术文档生成后再调用 subagent 测试
  - 覆盖正常路径、边界情况、异常路径
- 更新必要的 `__init__.py`（导出公共接口以及提供注释）
- **禁止将代码输出到对话框或写入文档**

## 阶段 6：技术学习文档

> 核心认知：覆盖规划时不要带着合规模式思维，必须以读者角度思考，确保技术文档内容质量极高，兼顾深度和广度

- 生成 `.project/docs/task_X.X/` 下的技术文档（按当前 Task 编号创建子文件夹）
- 按照 `project_info/tech_doc_design_spec.md` 编写

### 生成节奏

1. **覆盖规划**：在对话中声明文档清单和覆盖范围。**若 design.md 有架构决策章节，每个决策至少被一篇文档覆盖；否则覆盖范围仅限本 Task 的非决策知识点**。最小示例：
   ```
   领航员：整体架构认知（覆盖 D1↔D3 连锁、隐含前提）
   锚点1：Protocol vs ABC（锚定决策 2，展开结构子类型体系）
   锚点2：工厂函数与配置管理（锚定决策 3+4，展开工厂模式演进）
   非决策知识: 本 Task 中未被架构决策覆盖的可讲知识点
   ```
   - 对锚点文档：用三个触发点（连锁效应痕迹、隐含前提反推、反直觉行为）在源码中寻找深度入口。
   - **对非决策知识点**：深度的替代来源是——这个知识点如果不讲清楚，读者会在本 Task 的什么具体场景下犯错？从犯错场景反推必须讲透的内容。
2. **逐篇生成**：建议先写领航员再写锚点（先建全局框架，避免领航员退化成摘要拼接），但可自行判断方式
3. **深度追问**：写完核心技术解释后、写完全文后各追问一次——追问缺口、接引逃逸、标签伪装；声明任一触发点缺失前必须走三级防逃逸（深度复核 / 主题适配判断 / 禁止线）。详见 `tech_doc_design_spec.md` 深度协议
