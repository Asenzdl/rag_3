# Task 执行流程规范

> 触发条件：当用户说 **"开始 Task X.X"** 时，读取本文件并按流程执行
> 执行前必须先读取 CONTEXT_INDEX.md 确认当前 Task 状态和路径

---

## 总览：三阶段 + 审核模式

```
阶段1 反思驱动架构设计 → 阶段2 完整代码实现 → 阶段3 技术学习文档
```

审核模式控制阶段间的暂停节奏（用户在触发时指定）：

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| **模式 A**（默认） | 阶段1+2 连续执行 → 暂停确认 → 阶段3 分轮生成 | 平衡效率与质量 |
| **模式 B** | 每个阶段完成后暂停确认 | 想逐阶段审查 |
| **模式 C** | 阶段1+2 连续执行 → 暂停确认 → 阶段3 分轮生成 + 完成后调用 code-reviewer 审查 | 最高质量保障 |

触发示例：`开始 Task 1.5` → 模式A | `开始 Task 1.5 模式B` → 模式B | `开始 Task 1.5 模式C` → 模式C

## 阶段 1：反思驱动架构设计

### 输入

- 当前 Task 的 outline 文件（`.project_outline/phase_X/task_X.X_*.md`）
- 查看当前 Task 明确需要支持的 TODO（来源于 `.project_todo.md`）
- 必须严格按照 `project_info/task_doc_design_spec.md` 中的 **design.md 模板** 编写
- 按标题判断需读取的前后置 Task（按需加载，用于前瞻性设计和指出过往设计的不足）

### 过程（AI 内部反思，严禁生成代码文件）

1. **理解需求**：读取原始 task 需求，理解任务目标、面试知识点、生产级注意事项、验收标准
2. **构思初始方案**：设计模块结构、核心接口、数据流
3. **反思检查**（按顺序，每步不通过则迭代优化方案）：
   - **面试知识点全覆盖？** — 检查 task 中列出的每个面试知识点，确认架构中有对应的设计体现
   - **生产级注意事项全覆盖？** — 检查 task 中的每条注意事项（异常处理、配置管理、性能等），确认架构中有对应方案
   - **10维最佳实践充分落地？** — 逐维检查（参见下方清单），确认每个维度在本 Task 中有具体、可执行的落地方式或给出豁免声明
4. **整理最终架构设计**：确保决策有理有据，骨架代码达到"施工图纸"精度

### 10维最佳实践逐维检查清单

| 维度 | 检查要点 |
|------|----------|
| 1. 模块分离 | 文件职责是否单一？是否有"该模块包含/不包含"的明确边界？ |
| 2. 架构分层 | 数据流是否清晰？层间是否通过接口交互而非直接耦合？ |
| 3. SOLID 原则 | 是否遵循单一职责、开闭原则？**依赖是否倒置**（注入抽象而非硬编码）？ |
| 4. 封装与抽象 | 内部实现是否隐藏？公共 API 是否简洁？ |
| 5. 设计模式 | 是否选择了合适的模式（策略/工厂/适配器等）？**是否过度设计**？ |
| 6. 可观测性 | 关键操作是否有结构化日志？日志级别是否合理？是否包含上下文字段？ |
| 7. 配置管理 | 可调参数是否外部化？是否有合理默认值？是否禁止硬编码？ |
| 8. 鲁棒性/容错 | 异常是否分层（自定义异常体系）？边界情况是否处理？是否有明确的错误信息？ |
| 9. 可测试性 | 核心逻辑是否可 Mock？是否有纯函数可独立测试？ |
| 10. 可扩展性 | 是否预留了扩展点？是否为后续 Task 衔接留了 TODO？ |

### 产出

`.project_tasks/phase_X/task_X.X_design.md`

- **严禁包含**：完整实现代码、测试代码（这些在阶段2直接生成文件）
- 此文档的核心目标：**让用户有能力独立编写代码（宇宙第一重要）**（"施工图纸"精度的骨架中文注释步骤，不要具体的代码！）

## 阶段 2：完整代码实现

- 直接生成/修改 `src/` 下的代码文件
- 直接生成 `tests/` 下的测试文件
  - 覆盖正常路径、边界情况、异常路径
- 更新必要的 `__init__.py`（导出公共接口）
- **禁止将代码输出到对话框或写入文档**

---

## 阶段 3：技术学习文档

- 生成 `docs/task_X.X/` 下的技术文档（按当前 Task 编号创建子文件夹）
- 严格按照 `project_info/tech_doc_design_spec.md` 中的 **技术文档模板 + 深度标准** 编写
- 面向用户学习，由主 AI 生成，质量要求：
  - 技术原理要**深入本质**，不要浅尝辄止
  - 方案对比要有**明确的选择指南**
  - 代码示例必须**完整可理解**
  - 最佳实践要有**推荐做法 + 反模式**对比
  - **渐进式建构**：复杂技术需展示从简到繁的演进过程（如 V0→V1→V2），而非直接给出最终结果
  - **失败模式分析**：必须覆盖该技术的典型失败场景、边界情况和诊断方法
  - **理论解释**：不能停留在"怎么用"，必须解释"为什么有效"（底层机制/数学直觉）

### 分轮生成机制

1. **规划轮**：列出本 Task 需要生成的所有技术文档清单（文件名 + 主题 + 预计深度要点）
2. **逐篇生成**：每轮只生成**一篇**技术文档，生成后立即执行深度自检（按 `project_info/tech_doc_design_spec.md` 中的自检问题）
3. **自检通过**后再生成下一篇；自检不通过则当场修复
4. **全量复查**：所有文档生成完毕后，对整批文档做一次批量深度自检
