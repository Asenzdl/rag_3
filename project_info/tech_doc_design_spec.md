# 技术文档规范 v2

> **核心思路**：从"覆盖驱动"到"推理驱动"——不检查写了什么，而是确保想清楚了什么。覆盖问题在**生成前规划阶段**一次性解决，深度问题通过**触发协议**激发，文档本体不设必填要素清单。

---

## 一、文档类型

一个 Task 产出的技术文档由两类文档组成：

### 领航员文档

- **目标**：让读者获得系统级洞察——跨决策的连锁效应、架构的隐含前提
- **定位**：读者读完全部锚点文档后仍然会缺的那块拼图
- **一句话检验**：如果删掉领航员，把锚点文档全读一遍，读者是否能理解"各决策之间怎么互相约束、整体架构在什么条件下会崩"？如果不能 → 领航员有独立价值
- **通常一个 Task 一篇**

### 锚点文档

- **目标**：从一个架构决策出发展开一个知识领域，讲透这个领域本身
- **定位**：决策是入口不是边界——不只回答"项目为什么选这个"，还要回答"这个东西本身是什么、体系是什么"
- **一句话检验**：如果把"怎么用"的部分去掉，文档是否还有独立价值？如果没有 → 缺乏深度
- **一个架构决策通常对应一篇锚点，但多个紧密耦合的决策可以合为一篇**

---

## 二、生成前覆盖规划

文档本体不设必填要素清单——所有覆盖问题在**生成前规划阶段**一次性解决。规划在对话中声明（不写入文档），目的是让每个该被讲的内容进入某篇文档，**不是**让每篇文档按固定结构产出。

规划阶段未澄清的问题，不要在写作时凑——写作时的职责是把规划里定下的内容讲透，不是补全覆盖。

### 规划要回答的问题

**覆盖范围**（两类都要覆盖）：

- 决策类：design.md 中每个决策由哪篇文档锚定？
- 非决策类：本 Task 还产生了哪些值得展开的知识点（某个领域的体系化理解、某个反直觉机制、某个跨决策的连锁）？归到哪篇文档？

**领航员的独立价值**（领航员必须能回答）：

- 本 Task 的决策之间是否存在值得展开的连锁效应？是哪几对？
- 整体架构有哪些隐含前提？这些前提在什么条件下会崩？
- 如果只有锚点没有领航员，读者会缺哪块拼图？

**锚点的领域深度**（每篇锚点必须能回答）：

- 这篇锚点锚定哪个/哪些决策？
- 该决策对应的知识领域本身是什么？除了项目选型，还有哪些值得讲的底层机制、方案全景、失效边界？

### 附录模板（文档末尾机械拼接，不承担叙事）

- **代码映射**：模块/技术概念 → 代码位置的语义映射表（不是文件结构复述）
- **知识地图**：本文覆盖了什么、没覆盖什么、延伸方向。前置依赖只标非显而易见的（"需先掌握 C3 线性化"需要标注，"本文"不需要）

## 三、深度底线

### 核心原则

**不要用结论替代推理过程。** 以下都是这条原则的特例：

- "类型安全"后面必须跟具体的解释 → 不能用标签替代解释
- "选 Protocol 因为更灵活"后面必须跟灵活在哪、代价是什么 → 不能只给结论
- 方案对比表后面必须跟"在什么条件下选哪个" → 不能只列选项不给选择指南
- 失效描述后面必须跟诊断方法和解决方案 → 不能只说"会出问题"就收了

### 接引规则

接引（"方向关键词：XXX"）用于"本文讲了 L1，但 L2/L3 需要独立篇幅"的场景。写接引前先自问：我能用 3-5 句话讲到 L2 吗？如果能 → 直接讲，不要接引。

## 四、深度协议

### 触发点：从微观事实到宏观判断

不管是连锁效应、隐含前提还是深层洞察，路径都一样：**具体微观事实 → 一个"为什么"的停顿 → 反推到架构级判断**。如果你从抽象概念直接下笔，八成出表面关联。

**允许缺失**：以下三个触发点若在你的主题里不成立（决策没留下签名痕迹、失效追不到隐含前提、准备过程没撞到反直觉行为），对应分析允许缺失——但声明"缺失"前必须先走一遍下面的**三级防逃逸**。

**触发点演示：连锁效应**

写 `create_retriever` 的入口介绍时，我注意到它的返回类型标注是 `RetrieverProtocol`，不是 `VectorRetriever`。这个具体事实让我停下来问"为什么是抽象不是具体"——答案是决策 2。连锁效应就这样冒出来了：不是从"请展开连锁效应"这句规范里写出来的，是从一个签名细节反推出来的。如果只按规范从抽象出发，容易写出"决策 1 和 3 都用了 Settings"这种表面关联——共用输入不是连锁效应，A 改变 B 的形态才是。

**动作**：拿 design.md 中关键决策去源码里找它留下的具体痕迹——签名、返回类型、命名、字段标注。痕迹就是连锁效应的入口；找不到痕迹的决策，可能根本不在连锁链上。

**触发点演示：隐含前提**

写失效边界时，我发现了工厂函数的缓存冲突——改了 `.env` 后工厂仍返回旧实例。追问"为什么会冲突"，答案不是"缓存没失效"，而是"设计隐含了配置在进程生命周期内不变的假设"。隐含前提是从失效的根因反推出来的，不是凭空想出来的。

**动作**：写完失效边界后，对出现的失效追问"这个失效之所以可能发生，是因为设计默认了什么条件成立？"——默认条件就是隐含前提。

**触发点演示：深层洞察**

写 Protocol vs ABC 时，我注意到 `@runtime_checkable` 的 isinstance 检查只看方法名不看签名——一个 `invoke(self, x: int) -> str` 也满足 `isinstance(obj, RetrieverProtocol)`。这个反直觉事实是深层洞察的来源：Protocol 的类型安全完全依赖静态检查器，运行时是裸奔的。洞察不是从"请写一个 L3 段落"冒出来的，是从一个反直觉的代码行为冒出来的。

**动作**：读源码时留意反直觉的行为——一个方法/参数/返回值看起来该做 A 却做了 B，或者看起来该检查 X 却没检查。反直觉的地方通常是底层机制暴露的窗口。

### 三级防逃逸

"允许缺失"不是免责通道。声明某个触发点缺失前，按顺序走三级检查：

1. **深度复核**：是不是分析深度不够？比如"决策找不到签名痕迹"——是真的没留下，还是只看了入口函数没看内部调用链？回到源码多读一轮
2. **主题适配判断**：确认主题本身不具备该结构（如一个纯工具函数的锚点文档天然没有架构级连锁效应），可以接受对应段落缺失
3. **禁止线**：禁止为满足"应当有"而构造——宁可缺失，不要表面洞察（"假设网络可用"这种技术正确但无关痛痒的凑数，或"决策 1 和 3 都用了 Settings"这种共用输入伪连锁）

### 写后追问

写完核心技术解释后、写完全文后，各追问一次：

1. **追问缺口**：有没有一个读者看完会问"为什么"的地方，但我没解释？
2. **接引逃逸**：是不是用了接引跳过了一个我其实能讲清楚的点？如果能用 3-5 句话讲清楚 → 直接讲
3. **标签伪装**：如果删掉所有标签词（"类型安全"、"快速失败"），只保留解释性文字，读者还能看懂吗？

任一回答为"是"即需当场修改。

## 五、生成建议

### 顺序

建议先写领航员再写锚点。原因：先写锚点后，领航员容易退化为"锚点摘要拼接"——全局视角是事后补的不是先建的，连锁效应和隐含前提的发现依赖先有全局框架。但如果某个主题的锚点你特别有感觉想先写，也可以，只要领航员不是摘要拼接就行。

### 叙事手法

不规定叙事手法。以下几种都能产出深度，根据主题选择最自然的：

- **渐进式**（V0→V1→V2）：适合有明显演进路径的主题（如工厂函数从简单到复杂）。每一步的不足自然推动下一步的追问，深度自动冒出来
- **对比式**（A 哲学 vs B 哲学）：适合有清晰对立面的主题（如 Protocol vs ABC）。两边展开后差异自明，选择结论水到渠成
- **问题驱动式**（痛点→方案→为什么有效）：适合从运维经验出发的主题（如配置管理）。痛点够具体时，方案的解释自然有说服力

判断标准：哪种手法让你写着最不别扭，就用哪种。写着别扭通常意味着叙事线和主题不匹配。

## 六、领航员与锚点的分工

| 维度 | 领航员 | 锚点 |
|------|--------|------|
| 使命 | 系统级洞察：跨决策连锁效应、架构张力 | 领域级深度：一个知识领域的体系化展开 |
| 叙事重心 | 决策之间的**关系**和架构的**隐含前提** | 技术本身的**底层机制** |
| 决策处理 | **展开讲连锁效应**——决策 A 如何约束决策 B | 锚定一个决策，深度展开其背后的知识领域 |
| 失效视角 | 架构级——模块协作的断裂点、设计前提的适用范围 | 技术级——某个技术的失效边界 |

### 读者阅读顺序

1. 先读领航员 → 获得系统级洞察（决策之间怎么互相约束）
2. 再读感兴趣的锚点 → 深入特定领域
3. 需要编码时回到 design.md → 获得施工细节

## 七、文档头

每篇文档开头必须声明：

```markdown
> **关联 Task**：Task X.X [任务名称]
> **文档类型**：领航员 / 锚点
> **锚定决策**：[锚点文档填写——锚定 design.md 中的哪些决策]
> **覆盖的 Task 知识点**：[声明本篇覆盖的非决策类知识点]
> **关联文档**：[同 Task 的其他文档链接]
```

生成前必须做覆盖规划（在对话中声明，不写入文档）——这是整个体系**唯一的覆盖护栏**，规范本体不设必填要素。规划内容见第二节。
