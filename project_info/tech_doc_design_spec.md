# 技术学习文档设计规范 v3

> **适用范围**：所有 Phase 的 Task 阶段3 技术学习文档产出物
> **核心原则**：技术文档不是 API 文档翻译，而是用渐进式叙事让读者获得"原来如此"的洞察——理解"为什么这样设计"和"什么情况下会失败"
> **关联规范**：`project_info/task_doc_design_spec.md`（design.md 规范）

---

## 〇、核心理念

### 叙事线驱动

技术文档不是 10 道填空题，而是一条叙事线：

> **问题 → 原理 → 选型 → 实现 → 失效**

这条线规定了"先讲什么、后讲什么"，AI 不再逐章填写，而是沿着叙事线渐进建构。量化论据自然嵌入原理和选型章节，而非独立成章。

### 知识驱动源：design.md 的架构决策

每篇技术文档的核心主题，由 design.md 中的架构决策反向驱动。AI 在规划轮自主判断哪些决策共享一个技术主题、哪些决策独立成篇，但必须声明覆盖范围，确保无遗漏。

### 深度来自洞察，不来自章节

模板的职责是搭好舞台（叙事线清晰、章节精简、边界明确），给 AI 足够的开放空间产出洞察。过度约束会杀死惊喜，但叙事线本身就是约束——它规定了节奏，也留白了深度。

~~~markdown
## 一、核心问题

[为什么需要这个技术？解决什么问题？]

> **深度要求**：必须给出具体痛点 + 可感知的示例（不是泛泛而谈）
> **子问题数量**：建议 2~3 个，每个有具体示例
> **叙事角色**：叙事线的起点——让读者感受到"确实需要这个东西"

### 1.1 [子问题/挑战 1]
[说明 + 具体示例]

### 1.2 [子问题/挑战 2]
[说明 + 具体示例]

## 二、技术原理

[详细的算法/原理说明，含公式、图示、流程图]

> **深度要求**：必须解释"为什么有效"（底层机制/数学直觉），不能只说"怎么用"
> **渐进式建构**：算法/配置/Prompt 类主题需展示 V0→V1→V2 演进过程，每步说明改因+效果
> **适用性说明**：工具/框架类主题（如 structlog 配置）可用"核心用法 + 进阶技巧"替代渐进式建构
> **量化自然嵌入**：解释原理时，论据须有量化支撑（如复杂度分析、token 预算推算），数据须标注来源
> **Mermaid 图要求**：涉及数据流或 3+ 组件交互的主题，必须至少包含一个 Mermaid 流程图
> **叙事角色**：叙事线的核心——让读者获得"原来如此"的洞察

### 2.1 [核心概念]
[深入解释，不要浅尝辄止]

### 2.2 [关键机制]
[配合代码片段说明]

## 三、方案选型

[有哪些路径？为什么选这条？]

> **适用条件**：仅当主题存在 2+ 个业界主流方案时才要求完整方案对比。

## 四、实现要点

[用于教学的代码示例（说明原理/用法，不是复制实现代码） + 详细中文注释]

> **叙事角色**：叙事线的落地——让读者从理论走向实践

## 五、失效边界

[什么情况下会翻车？]

> **必选章节**，不可省略。每个技术主题至少覆盖 2 种典型失效场景。
> **四要素完整性要求**：每种失效模式必须包含完整的"症状→原因→诊断方法→解决方案"链。
> 缺少"诊断方法"是最常见的质量问题——"诊断方法"应回答"如何量化/确认问题已发生"。
> **与 design.md 的区别**：design.md 的"常见坑点"是编码时易犯的错，本章节是运行时/生产环境的失效边界。
> **叙事角色**：叙事线的终点——让读者知道技术的边界在哪
~~~

## 二、与 design.md 的内容边界

> 技术文档与 design.md 定位不同，应避免信息重叠，采用"互补+引用"策略。

| 知识类型 | design.md 负责 | 技术文档负责 | 重叠时处理 |
|---------|---------------|-------------|-----------|
| 选型结论 | 本项目选了什么+为什么 | 业界全景对比+通用选择指南 | 技术文档引用 design.md 的结论 |
| 编码级陷阱 | 常见坑点（写代码时易犯的错） | 失效边界（运行时的失效边界） | 按编码时/运行时分层 |
| 代码组织 | 文件结构+契约速览 | 技术概念→代码位置的语义映射 | 技术文档不重复文件结构 |
| 异常处理 | 异常捕获表+包装策略 | 技术失效的根因+诊断方法 | design.md 写"怎么捕"，技术文档写"为什么抛" |

## 三、深度标准

### 深度层级

| 层级 | 描述 | 示例 |
|------|------|------|
| L1 浅层 | 告诉你"怎么用" | "MMR 通过 λ 参数平衡相关性和多样性" |
| L2 中层 | 告诉你"为什么这样设计" | "λ=1 退化为纯相似度，λ=0 退化为纯多样性，实际使用 0.5-0.7" |
| L3 深层 | 告诉你"底层机制是什么"、"什么情况下会失败" | "MMR 分数 = λ·sim(q,d) - (1-λ)·max(sim(d,d'))，前半项拉向查询，后半项推离已选文档；当嵌入空间各向异性时，后半项可能选出不相关文档" |

**目标**：每个技术主题的叙事章节至少达到 L2，至少一个章节达到 L3。

### 文档长度指引

| 维度 | 建议 |
|------|------|
| 叙事章节每节下限 | 300~500 字 |
| 全文建议长度 | 2000~4000 字（不含代码块） |
| 超出上限时 | 拆分为多篇文档，每篇聚焦一个子主题 |

### 各章节最低深度要求

| 章节 | 最低深度 | 浅层写法（❌） | 深层写法（✅） |
|------|---------|--------------|-------------|
| 核心问题 | 痛点 + 具体示例 | "需要提升检索质量" | "中文查询英文文档时，BM25 关键词匹配完全失效（例：'部署' 查不到 'deploy'）" |
| 技术原理 | 底层机制/数学直觉 | "Few-shot 让 LLM 学会格式" | "Transformer 注意力检索最相似示例并补全格式，靠后示例权重更高（位置编码）" |
| 方案选型 | 优缺点 + 选择指南 + 量化权衡（如适用） | "方案A更好" | "方案A：O(1)扩展但牺牲类型安全；超过10个版本建议迁移方案C" |
| 实现要点 | 渐进式建构（算法/Prompt类）或 核心用法+进阶技巧（工具/框架类） + ✅/❌标注 | 直接贴最终代码 | 从最简版本开始，逐步演进到完整版本 |
| 失效边界 | 至少 2 种：症状→原因→诊断→解决（四要素完整） | 无此章节 或 缺诊断方法 | "长上下文指令淹没：症状=LLM忽略指令，原因=注意力稀释，诊断=中间指令遵从度<30%，解决=关键指令移至首尾" |
| 代码映射 | 技术概念→代码位置的映射表 | 无此附录 | "Prompt 版本管理 → src/generation/prompts.py:PROMPT_REGISTRY" |
| 面试要点 | 3~4 个考点 + 参考答案方向 | 无此附录 | "指数退避的数学原理 → 等待时间 = multiplier * 2^(attempt-1)" |

### 量化数据来源标注规范

量化论据自然嵌入叙事章节时，每个数据点须标注来源类型：

| 标注 | 含义 | 可信度 |
|------|------|--------|
| `[实测]` | 基于实际运行代码的测量结果 | 最高 |
| `[计算]` | 基于理论公式或代码逻辑的推算 | 高 |
| `[估算]` | 基于经验的合理估计 | 中 |
| `[引用:来源]` | 引自外部文献或官方文档 | 取决于来源 |

### 必选深度要素

| 要素 | 要求 | 适用条件 |
|------|------|--------|
| 理论解释 | 不仅"怎么用"，必须解释"为什么有效"（底层机制/数学直觉） | 所有主题 |
| 失效边界 | 至少 2 种：症状→原因→诊断→解决（四要素完整），诊断方法须可量化 | 所有主题 |
| 渐进式建构 | 展示从简到繁的演进（V0→V1→V2），每步说明改因+效果 | 算法/配置/Prompt类主题 |
| 核心用法+进阶技巧 | 展示基础用法 → 进阶配置 → 常见变体 | 工具/框架类主题 |
| 跨方案权衡 | 不仅列选项，必须给出"什么场景选什么"的决策指南 | 有方案选型的主题 |
| 代码映射 | 技术概念→代码位置的映射表 | 所有主题 |
| 面试要点 | 3~4 个考点 + 参考答案方向 | 所有主题 |
| Mermaid 流程图 | 涉及数据流或 3+ 组件交互时至少一个 | 数据流/多组件交互主题 |

### 严禁的浅层写法

```markdown
<!-- ❌ 只说结论不说理由 -->
选方案A因为更灵活。

<!-- ✅ 结论+理由+边界条件 -->
选方案A（字典映射）因为：(1) 新增版本只需加一行，O(1) 扩展；
(2) 但牺牲类型安全；(3) 超过10个版本建议迁移方案C。

<!-- ❌ 只说"怎么用"不说"为什么" -->
Few-shot 就是在 prompt 里加示例，LLM 就能学会格式。

<!-- ✅ 解释底层机制 -->
Few-shot 有效的根本原因：Transformer 注意力检索最相似示例并补全输出。
这意味着：(1) 示例相似度决定激活哪个；(2) 顺序影响权重；
(3) 超过注意力头数后边际收益递减。

<!-- ❌ 方案对比只有表格没有选择指南 -->
| 方案 | 优点 | 缺点 |
| A | 简单 | 不灵活 |
| B | 灵活 | 复杂 |

<!-- ✅ 表格+选择指南+量化权衡 -->
| 方案 | 扩展成本 | 类型安全 | 适用规模 |
| A 字典映射 | O(1) | ❌ | ≤10 版本 |
| B 类继承 | O(n) | ✅ | 任意 |
→ 选A：当前仅3个版本，扩展成本优势明显；后续超10版本时迁移C

<!-- ❌ 失效边界只写标题不写诊断链 -->
### 5.1 长上下文失败
Prompt 太长时 LLM 可能忽略中间指令。

<!-- ✅ 失效边界有完整诊断链（四要素缺一不可） -->
### 5.1 长上下文指令淹没
- 症状：LLM 忽略 System Prompt 中的格式指令，输出缺少引用标记
- 原因：注意力机制在长序列上稀释，中间 token 的平均注意力权重不足两端的 30%
- 诊断：对比有无中间 few-shot 时的指令遵从度，差值>40% 即为淹没
- 解决：将关键指令移至消息序列首尾（首因/近因效应），或精简中间内容

<!-- ❌ 量化论据无来源标注 -->
引用格式遵从率从 60% 提升到 90%。

<!-- ✅ 量化论据标注来源 -->
引用格式遵从率从 60% 提升到 90% [估算：基于 20 次手动测试]。
Token 消耗：V2 比 V1 增加约 150 tokens [计算：基于模板字符数估算]。

<!-- ❌ 常见问题是概念定义 -->
Q: 什么是 Few-shot？
A: Few-shot 是在 Prompt 中提供少量示例的技术。

<!-- ✅ 常见问题来自实际困惑 -->
Q: Few-shot 示例数量从 1 增加到 3，为什么效果提升不明显？
A: 因为 Transformer 注意力头的数量限制了同时关注的示例数。
1→2 个示例提升约 5%，2→3 个仅提升 1-2%，3+ 个边际收益递减且增加 token 消耗。
```

### 自检问题

写完技术文档后，必须逐条回答：

1. **去用测试**：如果把"怎么用"的部分去掉，文档是否还有独立价值？如果没有 → 缺乏深度
2. **决策测试**：读者能否根据文档理解"为什么项目选了这个方案"？如果不能 → 方案选型太浅
3. **诊断测试**：如果该技术在生产环境出了问题，读者能否根据文档诊断原因？如果不能 → 缺乏失效边界或诊断方法不具体
4. **洞察测试**：文档中是否有至少一个"原来如此"级别的洞察？如果没有 → 可能太浅
5. **映射测试**：读者能否根据文档快速找到项目中的对应代码？如果不能 → 缺乏代码映射
6. **面试测试**：文档中的面试要点是否覆盖了"为什么"和"什么情况下会失败"类问题？如果只有概念定义 → 面试要点太浅

## 四、生成流程

> **背景**：一口气生成多篇技术文档时，后生成的文档对深度要求的遵从度显著下降（Transformer 注意力稀释）。
> **解法**：规划轮 → 逐篇生成 + 自检。

### 规划轮

AI 自主判断每篇文档覆盖哪些设计决策，但在规划轮必须声明覆盖范围，确保无遗漏。

```markdown
## 技术文档规划

| # | 文件名 | 主题 | 叙事线 | 覆盖的设计决策 | L3 聚焦 |
|---|--------|------|--------|---------------|---------|
| 1 | xxx.md | ...  | 问题→原理→选型→实现→失效 | 决策1, 决策3 | [具体哪个叙事章节达到 L3] |
| 2 | yyy.md | ...  | 问题→原理→选型→实现→失效 | 决策2 | [具体哪个叙事章节达到 L3] |
```

> **遗漏检查**：规划完成后，对照 design.md 的决策列表，确认每个决策至少被一篇文档覆盖。

### 上下文锚点机制

首轮叙事章节生成完成后，在文档开头补充"上下文锚点"块，供后续附录生成时对齐：

```markdown
> **上下文锚点**（后续生成必须与此一致）
```

### 逐篇生成规则

1. 先生成 5 个叙事章节（核心问题 → 技术原理 → 方案选型 → 实现要点 → 失效边界），生成后立即执行深度自检
2. 自检不通过 → 当场修复 → 再次自检
3. 自检通过 → 输出简短结论（如"✅ 通过：去用/决策/诊断/洞察/映射/面试"） → 生成附录章节
4. 全部完成 → 全量复查

### 跨文档去重检查

全量复查时额外检查：
- 同一 Task 下的多篇文档是否有分析重复 → 保留在更相关的文档中，另一篇用"详见 xxx.md"引用
- 跨 Task 的文档是否有概念重复 → 在"前置知识"或"关联文档"中互相引用

## 五、质量检查清单

### 叙事章节必检项

- [ ] 明确标注关联 Task 和覆盖的设计决策
- [ ] 核心问题有具体痛点 + 可感知示例（非泛泛而谈）
- [ ] 技术原理达到 L2+（解释"为什么有效"而非仅"怎么用"），量化论据有来源标注
- [ ] 至少一个叙事章节达到 L3（底层机制/数学直觉/失效根因）
- [ ] 方案选型有选择指南 + 量化权衡（如适用），或设计选型说明
- [ ] 实现要点符合渐进式建构或核心用法+进阶技巧，✅/❌ 标注贯穿，代码示例符合最小可理解单元原则
- [ ] 失效边界至少 2 种，四要素完整（症状→原因→诊断→解决），诊断方法可量化，与 design.md 常见坑点不重复
- [ ] 涉及数据流/多组件交互的主题包含 Mermaid 流程图

### 附录章节检查项

- [ ] 代码映射表完整（技术概念→代码位置），不与 design.md 文件结构重复
- [ ] 面试要点 3~4 个，覆盖"为什么"和"什么情况下会失败"类问题
- [ ] 常见问题至少 3 个高质量 Q&A（来自实际困惑，非概念定义）

### 跨文档与边界检查项

- [ ] 前置知识和关联文档已标注，跨文档无内容重复
- [ ] 与 design.md 内容边界清晰（选型结论引用 design.md，失效边界聚焦运行时而非编码级）
- [ ] 深度自检通过：去用/决策/诊断/洞察/映射/面试
