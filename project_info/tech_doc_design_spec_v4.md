# 技术学习文档设计规范 v4

> **适用范围**：所有 Phase 的 Task 阶段3 技术学习文档产出物
> **关联规范**：`project_info/task_doc_design_spec.md`（design.md 规范）

---

## 〇、核心原则

**深度优先，广度其次，模板合规最后。**

宁可某个章节超出建议长度，也不要为了"叙事流畅"或"模板精简"而压缩深度。当模板合规与内容深度冲突时，深度优先。

技术文档的目标：让读者理解"为什么这样设计"和"什么情况下会失败"。如果去掉所有"怎么用"的部分后文档仍有独立价值，才算合格。

## 七、失效边界

[什么情况下会翻车？]

> **必选章节**，不可省略。每个技术主题至少覆盖 2 种典型失效场景。
> **四要素完整性**：症状→原因→诊断方法→解决方案，缺一不可。

> **与 design.md 的区别**：design.md 的"常见坑点"是编码时易犯的错，本章节是运行时/生产环境的失效边界。

## 二、深度标准

### 深度层级

| 层级 | 描述 | 示例 |
|------|------|------|
| L1 浅层 | 告诉你"怎么用" | "MMR 通过 λ 参数平衡相关性和多样性" |
| L2 中层 | 告诉你"为什么这样设计" | "λ=1 退化为纯相似度，λ=0 退化为纯多样性，实际使用 0.5-0.7" |
| L3 深层 | 告诉你"底层机制是什么"、"什么情况下会失败" | "MMR 分数 = λ·sim(q,d) - (1-λ)·max(sim(d,d'))，前半项拉向查询，后半项推离已选文档；当嵌入空间各向异性时，后半项可能选出不相关文档" |

**目标**：每个技术主题的核心章节至少达到 L2，至少一个章节达到 L3。

### 技术原理追问清单（强制）

对技术原理章节的每个核心机制，生成前必须逐条回答：

1. **是什么**：这个机制做了什么？（L1）
2. **为什么这样设计**：设计者的意图是什么？（L2）
3. **底层怎么实现**：源码/算法层面的实现机制？（L3——至少一个核心机制达到此级）
4. **边界条件在哪**：什么情况下这个机制会出问题或不按预期工作？
5. **与其他机制的交互**：当两个相关机制同时存在时，行为是什么？

**自检标准**：如果 3~5 都没有答案，说明深度不足，必须补充后再生成。

### 文档长度指引

| 维度 | 建议 |
|------|------|
| 技术原理子章节每节下限 | 300~500 字 |
| 全文建议长度 | 3000~5000 字（不含代码块） |
| 超出上限时 | 拆分为多篇文档，每篇聚焦一个子主题 |

### 量化数据来源标注

| 标注 | 含义 | 可信度 |
|------|------|--------|
| `[实测]` | 基于实际运行代码的测量结果 | 最高 |
| `[计算]` | 基于理论公式或代码逻辑的推算 | 高 |
| `[估算]` | 基于经验的合理估计 | 中 |
| `[引用:来源]` | 引自外部文献或官方文档 | 取决于来源 |

### 严禁的浅层写法

```markdown
<!-- ❌ 只说结论不说理由 -->
选方案A因为更灵活。

<!-- ✅ 结论+理由+边界条件 -->
选方案A（字典映射）因为：(1) 新增版本只需加一行，O(1) 扩展；
(2) 但牺牲类型安全；(3) 超过10个版本建议迁移方案C。

<!-- ❌ 只说"怎么用"不说"为什么" -->
Few-shot 就是在 prompt 里加示例，LLM 就能学会格式。

<!-- ✅ 解释底层机制 -->
Few-shot 有效的根本原因：Transformer 注意力检索最相似示例并补全输出。
这意味着：(1) 示例相似度决定激活哪个；(2) 顺序影响权重；
(3) 超过注意力头数后边际收益递减。

<!-- ❌ 量化论据无来源标注 -->
引用格式遵从率从 60% 提升到 90%。

<!-- ✅ 量化论据标注来源 -->
引用格式遵从率从 60% 提升到 90% [估算：基于 20 次手动测试]。
Token 消耗：V2 比 V1 增加约 150 tokens [计算：基于模板字符数估算]。

<!-- ❌ 失效边界诊断方法不可执行 -->
诊断：对比有无中间 few-shot 时的指令遵从度。

<!-- ✅ 诊断方法可执行 + 有判定标准 -->
诊断：
​```python
# 分别运行有无中间 few-shot 的 prompt，统计指令遵从次数
with_fewshot = count_compliance(run_prompt(prompt_v2_with_fewshot))
without_fewshot = count_compliance(run_prompt(prompt_v2_no_fewshot))
delta = (with_fewshot - without_fewshot) / without_fewshot
```
判定标准：delta > 0.4 即为指令淹没

<!-- ❌ 常见问题是概念定义 -->
Q: 什么是 Few-shot？
A: Few-shot 是在 Prompt 中提供少量示例的技术。

<!-- ✅ 常见问题来自实际困惑 -->
Q: Few-shot 示例数量从 1 增加到 3，为什么效果提升不明显？
A: 因为 Transformer 注意力头的数量限制了同时关注的示例数。
1→2 个示例提升约 5%，2→3 个仅提升 1-2%，3+ 个边际收益递减且增加 token 消耗。
```

## 三、与 design.md 的内容边界

| 知识类型 | design.md 负责 | 技术文档负责 | 重叠时处理 |
|---------|---------------|-------------|-----------|
| 选型结论 | 本项目选了什么+为什么 | 业界全景对比+通用选择指南 | 技术文档引用 design.md 的结论 |
| 编码级陷阱 | 常见坑点（写代码时易犯的错） | 失效边界（运行时的失效边界） | 按编码时/运行时分层 |
| 代码组织 | 文件结构+契约速览 | 技术概念→代码位置的语义映射 | 技术文档不重复文件结构 |
| 异常处理 | 异常捕获表+包装策略 | 技术失效的根因+诊断方法 | design.md 写"怎么捕"，技术文档写"为什么抛" |

## 四、生成流程

### 规划轮

AI 自主判断每篇文档覆盖哪些设计决策，但必须声明覆盖范围，确保无遗漏。

​```markdown
## 技术文档规划

| # | 文件名 | 主题 | 覆盖的设计决策 | L3 聚焦章节 | 预计量化数据点 |
|---|--------|------|---------------|------------|--------------|
| 1 | xxx.md | ...  | 决策1, 决策3 | [哪个章节达L3] | [至少2个] |
| 2 | yyy.md | ...  | 决策2 | [哪个章节达L3] | [至少2个] |
```

遗漏检查：对照 design.md 的决策列表，确认每个决策至少被一篇文档覆盖。

### 研究轮（新增）

逐篇生成前，对每篇文档必须执行以下至少 2 项研究动作：

1. **读源码/官方文档**：读取目标库的核心实现（如 pydantic-settings 的 `settings_customise_sources`），提取底层机制
2. **构造验证代码**：运行测试代码验证关键行为（如 `field_validator` before/after 差异、`frozen=True` 对 hash 的影响），产出 `[实测]` 数据
3. **搜索已知边界/故障**：查找 GitHub Issues、Stack Overflow 高票回答中的真实失效案例

研究产出以内部笔记记录（不写入文档），供生成轮引用。

### 逐篇生成规则

1. 每轮只生成一篇，生成后立即执行深度自检
2. 自检不通过 → 当场修复 → 再次自检
3. 自检通过 → 输出简短结论 → 下一篇
4. 全部完成 → 全量复查（深度一致性 + 跨文档去重）

### 深度自检六问

1. **去用测试**：去掉"怎么用"的部分，文档是否仍有独立价值？
2. **决策测试**：读者能否理解"为什么项目选了这个方案"？
3. **诊断测试**：生产环境出了问题，读者能否根据文档诊断？
4. **洞察测试**：文档中是否有至少一个"原来如此"级别的洞察？
5. **数据测试**：是否有至少 2 个带来源标注的量化数据点？
6. **因果测试**：每个叙事章节是否引用了前一个章节的结论作为起点？能否独立阅读而不需要前文？
