## Task 3.3 混合检索器（EnsembleRetriever：向量 + BM25 + RRF 融合）

### 任务目标
实现向量检索与 BM25 关键词检索的混合检索器，通过 Reciprocal Rank Fusion（RRF）算法融合两种检索结果，兼顾语义相似性和精确关键词匹配。

### 涉及文件
- `src/retrieval/ensemble.py`
- `src/retrieval/bm25_builder.py`（BM25 索引构建）

### 面试级知识点
- **向量检索 vs BM25 的互补性**：向量检索擅长语义相似性（"汽车"匹配"轿车"），但对精确术语匹配不敏感；BM25 擅长关键词匹配（"LangChain v0.3"精确匹配），但无法理解同义词和上下文。混合检索结合二者优势，是生产级 RAG 的标配。
- **RRF（Reciprocal Rank Fusion）算法原理**：`score(d) = Σ(1 / (k + rank_i(d)))`，其中 k 是常数（通常 60）。RRF 不依赖原始分数的量纲统一，比加权求和更鲁棒，因为它只依赖排名而非绝对分数。
- **EnsembleRetriever 的权重配置**：LangChain 的 EnsembleRetriever 支持 weights 参数，但 RRF 模式下权重作用有限。面试时要能说清 RRF 和加权求和的区别及适用场景。

### 生产级注意事项
- **BM25 索引的构建与更新**：BM25 检索器需要基于分块后的文档列表构建，存储在内存中。文档更新时需要重建索引。考虑在 `load_data.py` 入库后同时生成 BM25 索引并 pickle 保存，启动时加载。
- **分词器选择**：英文文档推荐使用 `rank_bm25` 的默认分词器（基于空格和标点），无需额外处理。中文场景则需要 jieba 分词，但本项目文档为英文，默认分词器足够。
- **检索数量配置**：向量检索和 BM25 各检索 `top_k` 个（如各 10 个），RRF 融合后取前 `final_k` 个（如 6 个）。两路检索数量应大于最终需要的数量，保证融合质量。
- **RRF 的 k 参数调优**：k=60 是业界常用值，但可根据数据集特点微调——k 越大，排名差异对分数影响越小，融合结果越平滑。在评估中用不同 k 值（20/60/100）对比。
- **混合检索的性能开销**：BM25 检索在内存中执行，毫秒级延迟；主要开销来自两路结果的 RRF 计算。整体延迟略高于单一向量检索但远低于 LLM 调用，性价比极高。

### 验收标准
- 实现 `EnsembleRetriever` 类，组合向量检索器和 BM25 检索器，通过 RRF 算法融合。
- 在 `load_data.py` 中增加 BM25 索引构建逻辑，索引保存到 `db/bm25_index.pkl`，检索时加载。
- 运行评估脚本，对比 4 种检索器（Base、MultiQuery、HyDE、Ensemble）的指标，生成对比表格。
- 手动测试精确术语查询（如"RunnablePassthrough"），验证 BM25 能命中向量检索可能遗漏的精确匹配文档。
- 检查 RRF 融合后的文档排名：打印前 3 个文档的来源，人工判断是否比单路检索更合理。
