# Task 执行工作流

---

## 流程总览

```
 0. 理解需求 → 1. 方案设计 → 2. 设计文档 → 3. 代码实现 → 4. 技术文档 → 5. 审查
```

---

## 步骤 0：理解需求

**前置**：必须理解透彻了用户画像和约束优先级

**动作**：理解 `.project/outline/phase_X_*/task_X.X.md` 中的需求，并与用户确认理解无误

**关键约束**：- **必须查官方文档**（用 langgraph 官方 MCP Docs by LangChain + context7 补充信息），避免因知识冻结导致的知识/概念陈旧以及 API 变更。

## 步骤 1：方案设计

**触发**：用户说"开始 Task X.X"

**动作**：启动 Plan agent，共同讨论设计方案

**目的**：为用户提供一个以官方文档为主、符合最佳实践的生产级设计方案


**流程**：
1. 主 agent 启动 Plan agent，附上需求上下文和搜索空间视角
2. Plan agent 展开搜索空间，产出候选方案对比和决策权衡分析
3. 主 agent 综合 Plan agent 的分析 + 自身判断，收敛到推荐方案
4. 产出 plan 文件（含候选方案对比和架构权衡，存入 `.claude/plans/`）
5. 主 agent 与用户确认推荐方案，展示推荐方案并收集用户反馈

**出口分支**：询问用户
- 同意推荐方案 → 进入步骤2
- 追问细节 → 继续讨论
- 要求优化 → 调整方案后重新确认

**Plan agent 硬性约束**：
- 角色设定：你是一名精通 LangChain、LangGraph 生态及生产级 RAG 系统设计的 AI 应用架构师，同时也是 Python 专家，和我（另一个架构师）一起讨论。
- 用户画像（关键约束）：[引用 CLAUDE.md]
- Task X.X 需求文档：[引用 task_X.X.md]

---

## 步骤 2：设计文档

**触发**：步骤 1 结束后，询问用户后声明"进入步骤2"

**动作**：根据收敛的决策，编写最终设计文档

**目的**：让用户在几个月后也能通过设计文档，透彻理解设计决策思路，以及通过施工图纸级别的代码蓝图自主实现代码

**流程**
1. 认真理解 `project_info/task_doc_design_spec.md` 中的要求
   - 确保理解了所有约束和要求，包括架构设计与权衡、代码蓝图章节等
2. 产出 `design.md`：主 agent 基于讨论收敛后产出的最终设计文档，存入 `.project/tasks/phase_X/task_X.X_design.md`

**design.md 关键要求**（引用 `project_info/task_doc_design_spec.md`）：
- 每个决策须有：候选对比 → 项目优势/硬伤 → 反驳推演 → 结论（含反转条件） → 反事实自检
- 非关键决策须记录，含排除理由
- 代码蓝图必须达到施工图纸的要求

**出口分支**：询问用户
- 设计文档符合要求 → 进入步骤3
- 设计文档不符合要求 → 继续讨论

---

## 步骤 3：代码实现

**触发**：步骤 2 结束后，询问用户是否进入步骤3

**动作**：直接写入 `src/` 和 `tests/`

**约束**：
- 同一文件的多个编辑合并为尽量少的操作
- 禁止将代码输出到对话框或写入文档（仅写入文件）
- 更新必要的 `__init__.py`（导出公共接口及提供注释）
- 测试失败超过 **2** 次后，**必须立即停止测试**，输出失败原因，

**出口分支**：询问用户
- 测试通过 | 测试失败 ≤ 2 次 → 进入步骤4
- 测试失败 > 2 次 → 是否继续测试
   - 继续测试 → 继续测试
   - 不继续 → 进入步骤4（但不修复测试代码）

---

## 步骤 4：技术学习文档

**触发**：步骤 3 结束后，询问用户是否进入步骤4

**动作**：理解 `project_info/tech_doc_design_spec.md` 后写技术学习文档，存入 `.project/docs/task_X.X/`（目录已构建）

**目的**：让用户在几个月后也能通过技术学习文档，理解知识体系

**子步骤**：

1. **覆盖规划**（在对话中声明，不写入文件）
   - 哪些决策对应哪些锚点文档
   - 领航员的独立价值（删掉领航员只读锚点，读者缺什么？）
   - 非决策知识点归属

2. **逐篇生成**
   - 建议先领航员再锚点（先建全局框架，避免领航员退化成摘要拼接）
   - 每篇文档按触发点驱动深度，非填空

3. **深度追问**
   - 全部写完后追问一轮：缺口、接引逃逸、标签伪装

**出口分支**：根据深度追问的问题，询问用户
- 修复 → 修复技术学习文档后**询问**是否进入步骤5
- 不修复 → **询问**是否进入步骤5

---

## 步骤 5：审查

**触发**：步骤 4 结束后，询问用户是否进入步骤5

**动作**：对设计文档 / 代码 / 技术文档进行客观审查并生成审查报告，存入 `.project/review/phase_X/task_X.X_review.md`
