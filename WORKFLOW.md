# Task 执行工作流（对话驱动版）

> 本工作流在 Task 2.5 验证通过，定位为参考节奏，非检查清单。
> 质量标准引用 `project_info/task_doc_design_spec.md` 和 `project_info/tech_doc_design_spec.md`，不重复定义。

---

## 节奏总览

5 步线性推进：

```
Phase A 设计 → Phase B 实现 → Phase C 文档 → Phase D（可选）评审
  │              │              │              │
  步骤1          步骤3          步骤4          步骤5
  方案设计        代码实现        技术文档        评审修复
  步骤2
  决策简读
```

---

## 步骤 1：方案设计

**触发**：用户说"开始 Task X.X"

**动作**：启动 Plan agent，共同讨论设计方案

**流程**：
1. 主 agent 启动 Plan agent，附上需求上下文和搜索空间视角
2. Plan agent 展开搜索空间，产出候选方案对比和架构权衡分析
3. 主 agent 综合 Plan agent 的分析 + 自身判断，收敛到推荐方案
4. 产出 plan 文件 + design.md

**关键约束**：
- 必须查官方文档（LangChain 任务用 Docs by LangChain MCP + context7 交叉验证）

1. 角色设定
2. 项目背景
3. 用户画像（关键约束）
4. 当前代码状态
8. 你的任务（产出要求）


```
请设计方案：[Task 名称]
你是一名**精通 LangChain、LangGraph 生态及生产级 RAG 系统设计的 AI 应用架构师，同时也是 Python 专家**，和我（另一个架构师）一起讨论。

项目背景 & 用户画像：[简述，引用 CLAUDE.md]

当前代码状态：
- 图拓扑：[当前图结构]
- 关键状态：[GraphState/GraphContext 字段]
- 数据流：[节点间的读写依赖]
- 现有设施：[已安装的依赖、可复用代码]

Task 需求：
- 验收约束：[关键条件]
- 已知设计陷阱：[提前告知常见坑]

需讨论的设计问题：
1. [问题 1]
2. [问题 2]
...

**硬性要求**：
- 必须查官方文档（LangChain 任务用 Docs by LangChain MCP + context7 交叉验证）

产出要求：
- 展开搜索空间后再收敛，不要一步到位
```

**产出**（两份）：
- plan 文件（Plan agent 的中间产出，含候选方案对比和架构权衡，存入 `.claude/plans/`）
- `design.md`（主 agent 基于讨论收敛后产出的最终设计文档，存入 `.project/tasks/phase_X/task_X.X_design.md`）

**design.md 质量标准**（引用 `project_info/task_doc_design_spec.md`）：
- 每个决策须有：候选对比 → 项目优势/硬伤 → 反驳推演 → 结论（含反转条件） → 反事实自检
- 非关键决策须记录，含排除理由
- 质量准则豁免逐个声明
- 代码蓝图达到"读者按注释可直接翻译为代码"的粒度

## 步骤 2：决策简读

**动作**：将设计中的关键拉扯点提炼为 3-5 句讲解

**格式**：

```
决策 X：选了 A 没选 B。
原因是：[一条本项目具体约束]。
如果某天这条约束变了（例如...），结论会反转。
```

**目的**：让用户看到拉扯过程，但不要求用户参与选择

**用户的三种反应方式**：

| 反应 | 处理 |
|------|------|
| 追问细节 | 展开讲解决策细节 |
| "知道了，继续" | 进入步骤 3 |
| "方向不对" | 回到步骤 1 迭代 |

---

## 步骤 3：代码实现

**动作**：直接写入 `src/` 和 `tests/`

**约束**：
- 同一文件的多个编辑合并为尽量少的操作
- 禁止将代码输出到对话框或写入文档（仅写入文件）
- 测试必须通过后才能进入下一步
- 更新必要的 `__init__.py`（导出公共接口及提供注释）

---

## 步骤 4：技术文档

**动作**：按 `project_info/tech_doc_design_spec.md` 标准写技术学习文档，存入 `.project/tasks/phase_X/`

**子步骤**：

1. **覆盖规划**（在对话中声明，不写入文件）
   - 哪些决策对应哪些锚点文档
   - 领航员的独立价值（删掉领航员只读锚点，读者缺什么？）
   - 非决策知识点归属

2. **逐篇生成**
   - 建议先领航员再锚点（先建全局框架，避免领航员退化成摘要拼接）
   - 每篇文档按触发点驱动深度，非填空

3. **深度追问**
   - 全部写完后追问一轮：缺口、接引逃逸、标签伪装
   - 追问后修复

---

## 步骤 5：可选评审

**触发**：用户要求，或步骤 3/4 涉及高风险判断

**动作**：

1. **自评**：对照 CLAUDE.md 质量准则 + 设计文档承诺逐维检查
2. **外部评审**：调用 subagent 从第三方视角审阅代码/文档
3. **合并发现**：写入评审文档（按严重度分级）
4. **逐项修复**：按优先级修复并验证

**不强制**。如果代码路径清晰、测试覆盖完整，可跳过。
