# Task 执行流程规范 v2 — 逐决策微循环

> 对 v1 的改造：将阶段 1 从"8 步线性"重构为"逐决策微循环"，匹配 AI 的认知模式。
> 阶段 2（代码实现）和阶段 3（技术文档）不变。

---

## 总览：三阶段 + 审核模式（不变）

```
阶段1 逐决策微循环架构设计 → 阶段2 完整代码实现 → 阶段3 技术学习文档
```

审核模式控制阶段间的暂停节奏（用户在触发时指定）：

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| **模式 A**（默认） | 阶段1+2 连续执行 → 暂停确认 → 阶段3 分轮生成 | 平衡效率与质量 |
| **模式 B** | 每个阶段完成后暂停确认 | 想逐阶段审查 |
| **模式 C** | 阶段1+2 连续执行 → 暂停确认 → 阶段3 分轮生成 + 完成后调用 code-reviewer 审查 | 最高质量保障 |

触发示例：`开始 Task 1.5` → 模式A | `开始 Task 1.5 模式B` → 模式B | `开始 Task 1.5 模式C` → 模式C

---

## 阶段 1：逐决策微循环架构设计

### 输入（不变）

- `project_info/CONTEXT_INDEX.md`（强制首读，用于路径映射和 Task 目标定位；若会话中已加载则复用）
- 当前 Task 的 outline 文件（`.project/outline/phase_X_*/task_X.X_*.md`）
- 按 `CONTEXT_INDEX.md` 中的 Task 目标判断需读取的 Task 文档（按需加载，用于前瞻性设计和指出过往设计的不足）

### 前置

进入本阶段时先一次性创建三个 Task（`addBlockedBy` 声明依赖链）：

```
TaskCreate: "阶段1 架构设计"        (独立)
TaskCreate: "阶段2 代码实现"        (addBlockedBy: 阶段1)
TaskCreate: "阶段3 技术文档"        (addBlockedBy: 阶段2)
```

这给 AI 提供了"后面还有任务"的外部记忆，在阶段 1 无限循环时产生收敛压力。

---

### 过程

#### 步骤 0：列出决策点（一次性）

在进入微循环前，显式输出本 Task 的所有决策问题清单。不写入文件，仅用于后续排序。

输出格式示例（在对话中输出，非 thinking）：

```
本 Task 的入口级决策（换方案会改代码结构）：
1. [决策名] — 一句话描述
2. [决策名] — 一句话描述

非入口级决策（归入 design.md 非关键决策章节）：
3. ...

前瞻性预留点（验收标准未要求，但可供用户选择）：
A. ...
B. ...
```

---

#### 微循环模板（每入口级决策独立执行一次）

```
┌─────────────────────────────────────────────────────────┐
│ 微循环 N：[决策名]                                       │
│                                                         │
│ 1. 陈述决策问题：一句话说清在决定什么                      │
│                                                         │
│ 2. 单轮分析：一次 thinking block，产出以下内容             │
│    a. 关键约束（影响此决策的项目现实，仅列特异性的）         │
│    b. 候选方案对比（≤ 2 个，每个写清本项目语境下的优劣势）    │
│    c. 反推验证（拿掉关键约束之一，结论是否反转）             │
│    d. 结论                                                │
│                                                         │
│ 3. 写入决策记录：.project/decisions/task_X.X_D{N}.md     │
│    格式见下方                                            │
│                                                         │
│ 4. 嵌入自检（写记录时一并完成）：                          │
│    □ 此决策覆盖了哪些 interview 知识点？                   │
│    □ 此决策覆盖了哪些生产级注意事项？                      │
│    □ 此决策是否与前面已写出的决策记录冲突？                 │
│                                                         │
│ 收敛约束：                                                │
│ - 每个微循环只允许一轮 thinking block                      │
│ - 一轮内无法收敛 → 标记为"待对齐"，在步骤 5 抛给用户        │
│ - 写入记录文件后，该决策关闭。后续微循环中不重新分析。        │
│   如发现新约束与已有记录冲突 → 更新记录（追加备注），不是重写  │
└─────────────────────────────────────────────────────────┘
```

##### 决策记录文件格式

```
---
问题：[一句话]
约束：[关键约束清单]
方案 A：[名称] — 优势 / 劣势
方案 B：[名称] — 优势 / 劣势
结论：选 [X]，理由 [一条扎根本项目现实的约束]
反转条件：[什么约束变成什么值时结论需重审]
状态：submitted
---

[可选] 分析备注：展开不能收入模板的推理细节
[可选] 更新备注：当后续微循环发现新约束后追加
```

##### 收敛约束的触发机制

- **一轮未收敛**：在记录文件中将 `状态` 设为 `pending_alignment`，不要在内部硬想
- **触发重新分析**：如果 AI 发现自己正在重新分析一个状态为 `submitted` 的决策 → 立即停止，检查是"忘记了"还是"新约束真的需要更新"。前者直接跳过；后者走更新流程

---

#### 步骤 5：一次性对齐（所有微循环完成后）

逐轮使用 AskUserQuestion。每一次询问一个**具体问题**：

1. **有分歧的决策**：步骤 0 标记的每个"待对齐"决策，一轮一个
2. **前瞻性预留点**：步骤 0 列出的每个预留点，一轮一个

一轮选完进下一轮。全部完成后进入步骤 6。

---

#### 步骤 6：合成 design.md

将所有决策记录组装为"架构决策与权衡"章节，然后补写：

- 模块结构
- 错误处理策略
- 测试策略
- 代码蓝图
- 常见坑点

蓝图暴露决策不自洽时 → 回步骤 6 修改（不进入微循环）。

**输出**：`.project/tasks/phase_X/task_X.X_design.md`

---

### 与 v1 的差异总览

| 维度 | v1 | v2 |
|------|----|----|
| 单位 | 8 步处理全部决策 | 每个决策独立微循环 |
| 决策记忆 | thinking 中隐式存在 | `.project/decisions/` 物理记录 |
| 需求对齐与卡点对齐 | 分两次 | 合并为一次性对齐 |
| 反思检查 | 独立一大步 | 嵌入每个微循环末尾 |
| 收敛控制 | 无（导致 6 次重复分析） | 每微循环限一轮，未收敛抛给用户 |
| TaskCreate | 每阶段创建一次 | 前置一次性创建三个，用 blockedBy 声明依赖 |

## 阶段 2：完整代码实现（不变）

> **生命周期**：进入时 Update(in_progress)，代码完成后 Update(completed)

- 直接生成/修改 `src/` 下的代码文件
- 直接生成 `tests/` 下的测试文件，必须在阶段3技术文档生成后再调用 subagent 测试
  - 覆盖正常路径、边界情况、异常路径
- 更新必要的 `__init__.py`（导出公共接口以及提供注释）
- **禁止将代码输出到对话框或写入文档**

---

## 阶段 3：技术学习文档（不变）

> **生命周期**：进入时 Update(in_progress)，文档完成后 Update(completed)

- 生成 `.project/docs/task_X.X/` 下的技术文档（按当前 Task 编号创建子文件夹）
- 按照 `project_info/tech_doc_design_spec.md` 编写

### 生成节奏

1. **覆盖规划**：在对话中声明文档清单和覆盖范围
2. **逐篇生成**：建议先写领航员再写锚点
3. **深度追问**：写完核心技术解释后、写完全文后各追问一次
