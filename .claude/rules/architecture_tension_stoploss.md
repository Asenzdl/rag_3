# 架构拉扯止损器

定位：用于暴露高风险架构误判，不追求形式化完备性。
     任何扩展建议须先证明当前版本漏掉了具体大错（举真实 Task 案例）。

触发判据：出现两个或多个合理方案指向不同方向的判断节点时启动。
典型信号：引入持有资源的新依赖 / 抽象层级选择 / 模块边界归属 / 多原则冲突。
不触发：明确 CRUD、纯算法、bug 修复、无设计空间的任务。

1. 各方方案/原则成立依赖什么前提？（对冲突双方均跑 Q1→Q2）
2. 这些前提在当前项目中是否真的成立？（给证据）
3. 生命周期和 ownership 是否匹配？（对高频盲区的强制重扫，不替代 Q1 对该维度的前提识别）
4. 错误会如何传播？（立即失败 / 静默累积）
5. 当前判断在什么条件下需要重审？
