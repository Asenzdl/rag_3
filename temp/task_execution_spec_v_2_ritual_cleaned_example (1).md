# Task 执行流程（收敛版）

> 目标：维持高密度架构推演，避免 AI 滑入 checklist / compliance cognition。
>
> 核心原则：
> - 推演优先于审计
> - 暴露 tradeoff，而不是证明“已充分考虑”
> - 暴露失败路径，而不是表演“全覆盖”
> - 高信息密度优先于结构完美

---

# 元规则（先读）

## 项目特异性底线

所有推演内容：

必须绑定当前项目上下文。

判据：

任何一句放到另一个项目里仍成立：

默认视为 ritual 化输出。

不通过。

每节写完后必须自问：

> 本节内容是否存在“放到别的项目仍成立”的泛化表达？

若存在，重写。

---

## 禁止出现的 audit / compliance 话语（示例，非穷尽）

- 是否充分考虑
- 是否全覆盖
- 是否生产级
- 是否遗漏
- 是否完全落地

这些话语会将推演拉入 compliance cognition。

禁止使用。

---

# 阶段 1：架构推演

> 本阶段目标：
> 不是证明“考虑全面”，而是暴露真正影响架构的拉扯、权衡与风险。

## 1. 需求重述

用自己的话明确：

- 用户真正目标
- 当前 Task 边界
- 非目标
- 成功标准

禁止 AI 自行补全隐含目标。

本节结束后：

必须向用户呈现一句话确认：

> “我理解的核心目标是 X，Y 视为非目标，对吗？”

即使 AI 自认为没有歧义，也必须确认。

目的：

避免后续所有高密度推演建立在错误目标之上。

本节写完后自问：

> 本节内容是否仍可直接套用于其他项目？若可以，重写。

---

## 2. 架构决策推演

仅当当前 Task 存在真实架构拉扯时，才展开完整推演。
小型 Task 可跳过本章，直接进入蓝图。

### 跳过条件

以下条件**必须同时满足**才允许跳过本章。任意一条不满足，即使感知不到拉扯，也必须至少执行"当前最大 tradeoff"、"动态失败情境"、"当前不确定性"三项：

- 不改变生命周期
- 不改变状态边界
- 不改变数据流
- 不新增异步/并发
- 不改变模块关系
- 不涉及新的 failure mode

---

### 当前方案

说明：

- 当前准备采用什么方案
- 为什么这个方向适合当前项目状态
- 当前项目哪些约束影响了选择

要求：

必须结合当前项目上下文。

禁止泛化描述。

本节写完后自问：

> 当前方案是否真正绑定当前项目状态？

---

### 当前最大 tradeoff

说明：

当前方案真正困难的拉扯点是什么。

重点：

- 为什么这里难
- 为什么两个方向都合理
- 当前项目哪个条件改变了最优解
- 为什么这里容易误判

禁止：

- 空泛对子
- 教科书 tradeoff
- 可迁移到任意项目的废话

runtime 校准：

> 本节中，哪条项目约束一旦移除，这个 tradeoff 就不成立了？

---

### Rejected Alternatives

至少说明一个真实替代方案。

要求：

- 为什么它在当前项目里不合适
- 放弃代价是什么
- 它在哪类项目反而更合理

重点不是“列方案”，而是暴露：

- 判断边界
- 适用条件
- 架构品味

本节写完后自问：

> 当前 rejected alternative 是否真的在当前 Task 中考虑过？

---

### 动态失败情境（主体）

现场构造 3 个候选失败场景。

要求：

- 当前 Task 特有
- 有压力
- 有信息缺失
- 有真实后果
- 有至少两个主体

然后：

选择其中最危险的一个展开。

展开时说明：

- 为什么它最危险
- 为什么容易误判
- 当前系统哪个位置最脆弱
- 哪个隐含假设会在这里失效

禁止：

- OOM
- 性能问题
- 可扩展性不足
- 可维护性下降

这种脱离上下文的抽象风险词。

runtime 校准：

> 这个场景放到另一个项目里，哪个细节会最先失效？

---

### Failure Mode（从动态情境中提炼）

从上面的动态失败情境中提炼：

- 当前系统最脆弱的位置
- 哪个 failure mode 最值得优先防御
- 为什么这个问题难被 tests 或 review 提前发现

Failure mode 不是抽象风险标签。

它必须来自当前情境推演，而不是独立填空。

---

### 当前不确定性

明确：

- 哪部分已确认
- 哪部分依赖经验判断
- 哪部分需要后续验证
- 哪部分缺少信息

目的：

防止 AI 把推测写成事实。

runtime 校准：

> 哪条判断只是在语境中显得合理，但其实没有任何可观测证据？

---

## 3. 蓝图输出

输出：

- 模块结构
- 生命周期
- 数据流
- 关键接口
- 状态流转
- 测试重点
- 每个关键接口的不确定性来源

要求：

让读者理解：

- 系统如何运转
- 为什么这样组织
- 哪部分最脆弱

而不是只看到“代码结构”。

要求：

每个关键接口必须标注：

- 当前不确定性来自哪里
- 对应哪个 tradeoff
- 缺少哪部分信息

禁止蓝图假装“所有问题已经确定”。

本节写完后自问：

> 蓝图是否偷偷绕过了前面的不确定性？

---

# 阶段 2：实现

> 本阶段目标：
> 将阶段 1 的架构推演落地。
>
> 不再主动发起新的架构哲学反思。
>
> 但以下情况必须回到阶段 1：
> - 当前设计无法落地
> - tradeoff 发生变化
> - 原假设失效
> - 生命周期冲突
> - 新 failure mode 出现

---

## 1. 小步实现

避免：

- 超大一次性生成
- 无法定位错误
- reviewer 无法审查

---

## 2. 保持架构一致性

实现过程中：

如果发现：

- 当前设计无法落地
- tradeoff 发生变化
- 原假设失效
- 生命周期冲突

必须回到阶段 1 修正。

禁止：

代码实现阶段偷偷改变架构。

---

## 3. Tests

至少覆盖：

- 核心路径
- 动态失败情境暴露出的 failure mode
- 关键边界
- 当前 Task 最脆弱部分

要求：

tests 必须显式绑定阶段 1 推演得到的 failure mode。

tests 不允许 AI 自主跳过。

---

## 4. Reviewer（关键 Task）

以下情况必须 reviewer：

- 生命周期变化
- 状态管理变化
- 数据模型变化
- 并发/异步变化
- 跨模块重构
- 架构决策变化

目的：

提供异质认知，而不是“证明没 bug”。

---

# 阶段 3：沉淀

> 本阶段目标：
> 沉淀真正有长期学习价值的认知。

禁止：

- “开发顺利完成”
- “架构合理”
- “整体较优”

这种成功学总结。

---

## 1. 技术文档

分为两类：

### 决策层（必写）

- 为什么这样设计
- 哪个 tradeoff 最关键
- 哪个隐含约束决定了架构
- 哪个 failure mode 最危险
- 哪个错误最容易误判
- 哪个地方最反直觉

### 知识层（按 Task 内容决定是否写）

例如：

- LangGraph API 行为
- Chroma 使用细节
- 生命周期坑点
- 框架限制
- 特殊调试经验

不要求 audit 式“全覆盖”。

只记录：

真正影响当前项目判断的知识。

不追求：

- 面面俱到
- 教科书化说明

---

## 2. Incident 残留

每个 Task 结束后：

记录一句：

```text
[Task X.Y]
真正危险的错误是什么？
为什么当时差点没发现？
```

如果是冷启动阶段（暂无历史 incident）：

记录：

```text
当前执行规范中，最容易让我开始“填空表演”的部分是什么？
```
```

要求：

- 极简
- 高信息密度
- 项目特异

目的：

积累真实 failure data。

---

# 最终目标

这套流程不是为了：

- 生成“看起来很认真”的设计文档
- 证明 AI 已充分考虑
- 完成 checklist

而是为了：

- 暴露真正影响架构的拉扯
- 暴露 failure mode
- 暴露判断边界
- 提升长期工程认知密度
- 让架构推演过程可学习、可讨论、可反驳
