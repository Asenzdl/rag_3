# Task 执行流程规范 v2

> 触发条件：当用户说 **"开始 Task X.X"** 时，读取本文件并按流程执行

---

## 总览：三阶段 + 审核模式

```
阶段1 反思驱动架构设计 → 阶段2 完整代码实现 → 阶段3 技术学习文档
```

审核模式控制阶段间的暂停节奏（用户在触发时指定）：

| 模式 | 行为 |
|------|------|
| **模式 A**（默认） | 阶段1+2 连续执行 → 暂停确认 → 阶段3 分轮生成 |
| **模式 B** | 每个阶段完成后暂停确认 |
| **模式 C** | 阶段1+2 连续执行 → 暂停确认 → 阶段3 分轮生成 + 完成后调用 code-reviewer 审查 |

触发示例：`开始 Task 1.5` → 模式A | `开始 Task 1.5 模式B` → 模式B |

---

### 全局前置：规范预读（每个 Task 启动时必做）

在进入阶段0之前，依次读取以下规范并完成阅读确认：

**步骤 1：范围声明**
- 读取 `task_doc_design_spec.md`，枚举其**全部章节标题**（含子章节），形成文档地图
- 在对话中输出地图（如 `本文档共 X 章：架构决策、非关键决策确认、已知后续替换、模块结构、错误处理、测试策略、代码蓝图、交互时序图、常见坑点、质量自检`）
- 目的：建立完整认知框架，防止按信息密度跳过章节

**步骤 2：蓝图约束前置确认**
- 读取 `task_doc_design_spec.md` 的「代码蓝图」章节，重点关注「为什么」四类触发条件：
  - 设计决策：有多种合理方案，选了其中一个
  - 反直觉辩护：行为和直觉相反
  - 功能取舍：做了 A 没做 B
  - 替代方案排除：读者可能想到另一种做法
- 确认：在做架构决策时，四类追问的视角必须作为决策质量的检验工具

**步骤 3：阅读模式声明**
- 确认当前为**读者模式/服务模式**而非**合规模式**
- 合规模式信号：只在核对"我是否满足每个字段要求"
- 读者模式信号：思考"这个产出对用户有什么学习价值"

**步骤 4：读取 `tech_doc_design_spec.md`**（仅声明确认已读，不展开）

## 阶段 1：架构设计

### 输入

- `CONTEXT_INDEX.md`
- 当前 Task 的 outline： Read `.project/outline/phase_X_*/task_X.Y_*.md`
- 按 `CONTEXT_INDEX.md` 中的 Task 目标判断需读取的 Task 文档（按需加载，用于前瞻性设计和指出过往设计的不足）

### 过程（AI 内部反思，严禁生成 `src/`、`tests/` 下的代码文件）

> 核心认知：必须思考用户画像，理解用户需求

1. **理解需求**：理解任务目标、面试知识点、生产级注意事项、验收标准

2. **构思初始方案**：设计模块结构、核心接口、数据流

3. **需求对齐**（构思完成后，逐轮使用 AskUserQuestion）：每个对同等有效方案的无法抉择独立作为一轮选择题展示，一轮选完再进下一轮；若无分歧则跳过
   - 每个选项是一个**具体方向**，不是抽象判断
   - 每道题附"其他"自定义入口（AskUserQuestion 自带）
   - 用户选完一道后，根据回复调整方案再展示下一道
   - 全部完成后进入步骤 4

4. **写出架构决策**：按 `task_doc_design_spec.md` 的"架构决策与权衡"章节格式，写出决策章节（仅决策，不含蓝图）。

   ---
   **强制前置检查（基于全局前置时建立的认知）**
   - **非关键决策**：当前 thinking 中是否有实质性对比段落但不命中入口判据？如有 → 记录到「非关键决策确认」章节
   - **已知后续替换**：是否有"现在 A、未来 B"且有硬信号（接口/生命周期/依赖方向变更）？如有 → 记录到「已知后续替换」章节
   - **蓝图"为什么"四类检验**：当前决策是否在任意一类下存在不自洽隐患？
     - 设计决策：反转条件是否写到了"具体值"精度？
     - 反直觉辩护：如果我需要在代码中写注释解释它，是否说明决策不自然？
     - 功能取舍：被否方案的"硬伤"是否在蓝图可观测的维度（返回值类型、异常声明）留下痕迹？
     - 替代方案排除：如果我写蓝图时想到"不，这里应该用方案 B"，是为什么？
   - **读者模式检验**：这个决策和一个入门学习者有什么关系？它能教会什么？

5. **反思检查**（对照已写出的决策，逐项通过或标记为卡点）：
   - **面试知识点全覆盖？** — 确认架构中有对应的设计体现或更优方案
   - **生产级注意事项全覆盖？** — 确认架构中有对应方案或更优方案
   - **10维最佳实践充分落地？** — 按 CLAUDE.md「质量准则」逐维检查。先判断是否适用，不适用则豁免；适用再检查落地方式
   - **前瞻性设计报告** — 报告验收标准外的潜在预留点选项，在卡点对齐中呈现
   - **非关键决策遗漏检查？** — 过一遍"本 Task 中做出的、不是入口级但可能让后续读者困惑的选择"
   - **卡点标记**：遇到架构决策且 3 轮内无法自行收敛时，创建待确认标记，继续后续检查

6. **决策迭代**：反思检查发现问题的 → 修改决策 → 重跑检查；无修改 → 决策稳定，进入下一步

7. **卡点对齐**：
   - 如果无卡点，跳过此步
   - 使用 AskUserQuestion 将每个卡点（含前瞻性设计报告中的预留点）作为选择题呈现
   - 用户确认后修正方案，卡点全部解决后进入下一步

8. **写出蓝图 + 其余章节**：基于稳定决策，写完剩余章节
   - 蓝图暴露决策不自洽之处时，回步骤 4 修改决策

### 代码蓝图粒度

按 `project_info/task_doc_design_spec.md` 中"代码蓝图"章节执行

### 输出

- **核心目标**：深刻理解所有核心架构决策，根据代码蓝图独立编写代码
- 存储于 `.project/tasks/phase_X/task_X.X_design.md` (目录已存在，直接生成新文件)

## 阶段 2：完整代码实现

> 对同一文件的多个编辑应合并为尽量少的操作。

- 直接生成/修改 `src/` 下的代码文件
- 直接生成 `tests/` 下的测试文件，必须在阶段3技术文档生成后再调用 subagent 测试
  - 覆盖正常路径、边界情况、异常路径
- 更新必要的 `__init__.py`（导出公共接口以及提供注释）
- **禁止将代码输出到对话框或写入文档**

## 阶段 3：技术学习文档

> 核心认知：覆盖规划时不要带着合规模式思维，必须以读者角度思考，确保技术文档内容质量极高，兼顾深度和广度

- 生成 `.project/docs/task_X.X/` 下的技术文档（按当前 Task 编号创建子文件夹）
- 按照 `project_info/tech_doc_design_spec.md` 编写

### 生成节奏

1. **覆盖规划**：在对话中声明文档清单和覆盖范围。**若 design.md 有架构决策章节，每个决策至少被一篇文档覆盖；否则覆盖范围仅限本 Task 的非决策知识点**。最小示例：
   ```
   领航员：整体架构认知（覆盖 D1↔D3 连锁、隐含前提）
   锚点1：Protocol vs ABC（锚定决策 2，展开结构子类型体系）
   锚点2：工厂函数与配置管理（锚定决策 3+4，展开工厂模式演进）
   非决策知识: 本 Task 中未被架构决策覆盖的可讲知识点
   ```
   - 对锚点文档：用三个触发点（连锁效应痕迹、隐含前提反推、反直觉行为）在源码中寻找深度入口。
   - **对非决策知识点**：深度的替代来源是——这个知识点如果不讲清楚，读者会在本 Task 的什么具体场景下犯错？从犯错场景反推必须讲透的内容。
2. **逐篇生成**：建议先写领航员再写锚点（先建全局框架，避免领航员退化成摘要拼接），但可自行判断方式
3. **深度追问**：写完核心技术解释后、写完全文后各追问一次——追问缺口、接引逃逸、标签伪装；声明任一触发点缺失前必须走三级防逃逸（深度复核 / 主题适配判断 / 禁止线）。详见 `tech_doc_design_spec.md` 深度协议

## 审核模式扩展（补充）

### 模式 B/C 的审核报告格式

当阶段完成时，不在对话中展开详细说明，而是按以下结构输出一份审核报告：

```markdown
---
phase: 1
task: "X.Y"
status: pending_approval
---

## 决策摘要
- **D1**：[名称] → 选 [方案A]
  - 反转条件：[具体约束→具体值]
  - 前提：[当前项目特有的一个事实]

## 非关键决策（2 条）
- [主题]：想过去选 [方案] 但不够格入主决策，因 [约束]

## 已知后续替换
- [接口/模块] → [Task Z] 切换，原因是 [硬信号类型]

## 拉扯点（学习价值）
| 拉扯点 | 选项 | 前提依赖 | 结论 |
|--------|------|---------|------|

## 需要确认
1. [ ] D1 的前提是否成立？
2. [ ] [其他无法自主收敛的选项]
```

审核报告**不替换** design.md 的完整输出——design.md 照常写入 `.project/tasks/`，审核报告是供你快速确认的摘要。

## 阶段间合规基准

### 阶段验收标准

| 维度 | 检查信号 | 违反后果 |
|------|---------|---------|
| 读者模式 | 文档开头未声明服务对象 | 回退到全局前置步骤3 |
| 范围声明 | 未枚举所有章节即进入阶段1 | 回退到全局前置步骤1 |
| 蓝图前置 | 决策记录未被四类追问检验 | 回退到步骤4 |
| 非关键决策 | 存在实质 thinking 对比但未记录 | 补写非关键决策确认章节 |
| 已知后续替换 | 有切换信号但未记录 | 补写已知后续替换章节 |
