# AI 约束设计：从理论地基到工程实践的完整体系

---

## 导言：本文档的起源与结构

本文档源于一场关于 AI 约束设计的深度讨论。讨论的起点是一个核心洞见：**AI 的原生能力是语义操作，因此任何用语言表达的约束都在 AI 的能力射程之内。** 这不是 bug，而是 feature——你用语言约束一个语言模型，就像用代码约束一个编译器：它比你更懂这个介质。

从这个起点出发，讨论经历了多轮迭代、批评、修正和深化，最终形成了一套从理论地基到工程实践的完整体系。本文档将完整记录这一体系，按逻辑演进顺序分为四个部分：

1.  **理论地基**：约束生效的根本原则、射程内/外的精确边界、完整的分类体系
2.  **三层结构**：规约层、控制层、基线层的正交关系
3.  **约束基元**：约束的最小完整单元及其三要素
4.  **工程实践**：需求转化、多约束组合、组织关系、优先级设计、约束膨胀处理

# 第一部分：理论地基

## 一、核心命题与根本原则

### 1.1 原始洞见

**核心事实**：AI 的原生能力就是语义操作。任何用语言表达的约束，都在 AI 的能力射程之内。

这不是 bug，是 feature——你用语言约束一个语言模型，就像用代码约束一个编译器：它比你更懂这个介质。

**从这个事实出发，约束设计的本质问题是**：如何让约束的生效机制落在 AI 能力射程之外？

### 1.2 射程概念的修正

经过深入讨论，原始表述中的"语义射程"需要修正为**"生成机制射程"**。

**理由**：LLM 的主导控制通道是语义操作（token-level + attention + distribution），但并非唯一通道。还包括：

-   位置编码（结构偏置）
-   采样策略（temperature / top-p）
-   解码约束（logit bias / stop tokens）

**修正后的定义**：  
**射程内** = 通过模型内部生成机制可调控的约束  
**射程外** = 脱离模型生成机制的约束

这一定义修正确保了我们能够解释为什么 logit bias 能硬控输出、为什么 stop sequence 是强约束（但不是语义）。

**更精确的表述**：  

> 射程内 = 穿透模型生成机制起效的约束，都会受制于模型对生成过程的控制力  
> 射程外 = 不依赖模型生成机制，由外部系统或工作流结构直接执行的约束

### 1.3 框架的本质

这个框架的本质不是"约束 AI 的三种方法"，而是揭示了**约束设计的唯一根本原则**：

> **用介质 A 去约束一个精通介质 A 的行为体，约束的有效性永远受制于对方对介质 A 的掌握程度。要获得不受制于此的约束力，必须把生效机制锚定在介质 A 之外。**

语言只是介质 A 的一个特例。如果未来 AI 精通了其他介质（比如直接操作机器代码、直接生成电路设计），同样的逻辑会复现。

### 1.4 概率控制与空间裁剪——射程内/外的本质区别

经过讨论，我们确认了射程内/外最根本的工程区别：

| 类型   | 本质                           | 保证程度                                      |
| ------ | ------------------------------ | --------------------------------------------- |
| 射程内 | **概率控制**——调整概率分布     | 永远无法提供 hard guarantee，只能降低违规概率 |
| 射程外 | **可行空间控制**——改变可行空间 | 可以提供 hard guarantee，但可能误伤正常输出   |

**这是工程决策的分水岭**：射程内手段永远只能降概率，射程外手段才能给保证。工程上，两者不是替代关系——射程内做日常行为塑造，射程外做关键风险兜底。

## 二、约束的完整分类体系

### 2.1 射程内约束（生成机制控制 = 概率控制）

射程内约束不是四类并列的分类，而是一个**三层控制结构**。四类机制是按**生效机制**划分的，三层结构是按**作用层级**划分的。两个维度正交互补：三层结构适合**设计约束策略**（在哪个层发力），四类机制适合**诊断失效模式**（失效是怎么发生的）。

#### 三层控制结构（按作用层级）

##### 1. 分布层（Distribution Shaping）

-   **本质**：改变 P(output | context)
-   **约束类型**：分布内化型、few-shot、角色设定
-   **生效原理**：激活隐式分布，使违规模式不被采样。不是"我判断这个合规"，而是"我根本没往那个方向想"
-   **被绕开方式**：分布缓慢漂移，被新的上下文覆盖

##### 2. 注意力层（Attention Steering）

-   **本质**：改变当前 step 的 token 权重分配
-   **约束类型**：注意力提醒型、显式规则
-   **生效原理**：改变局部注意力权重，在决策点拉高规则相关 token 的注意力
-   **被绕开方式**：注意力竞争中被更强信号淹没

##### 3. 轨迹层（Trajectory Constraint）

-   **本质**：对"生成序列的路径"施加约束，而非单点 token
-   **约束类型**：逻辑钳制型、格式强制型（AI 自维持）
-   **子类型详述**：
    -   **逻辑钳制型**（语义一致性压力）：利用自洽性压力，使违规代价是自相矛盾。AI 在回避自相矛盾上比遵守规则要难得多
    -   **格式强制型**（语法路径约束）：用输出骨架限制生成空间，压缩自由解释。内容填进固定槽位，偏离骨架的代价是破坏显式格式承诺
-   **被绕开方式**：
    -   逻辑钳制型：构建合理化叙事来消解矛盾感
    -   格式强制型：在槽位内部进行微观语义重构

#### 射程内四类约束机制（按生效机制）

| 类型         | 生效机制                             | 被绕开方式                   |
| ------------ | ------------------------------------ | ---------------------------- |
| 分布内化型   | 激活隐式分布，使违规不被采样         | 分布缓慢漂移，被新上下文覆盖 |
| 注意力提醒型 | 在决策点拉高规则相关 token 注意力    | 注意力竞争中被更强信号淹没   |
| 逻辑钳制型   | 利用自洽性压力，使违规代价是自相矛盾 | 构建合理化叙事消解矛盾感     |
| 格式强制型   | 用输出骨架限制生成空间               | 在槽位内部进行微观语义重构   |

**关键说明**：格式强制与结构判定的边界——格式强制是 AI 自己维持格式约束；如果格式校验由外部系统执行（如 JSON parser 报错），它就滑入了结构判定。边界在于**谁执行校验**。

---

### 2.2 射程外约束（外部系统控制 = 空间裁剪）

经过完整讨论，射程外约束共有**四类**：

| 类型               | 生效机制                                | 典型手段                                                     |
| ------------------ | --------------------------------------- | ------------------------------------------------------------ |
| **过程消除**       | 重构工作流，使需要判断的时刻不存在      | 硬路由、任务原子化、预定义分支                               |
| **生成时结构约束** | 外部系统在生成时动态裁剪可行 token 空间 | constrained decoding、grammar-based decoding、function calling schema enforcement |
| **结构判定**       | 后验二元事实核查，由外部系统执行判定    | 关键词熔断、正则表达式校验、形式验证、JSON schema 校验       |
| **委托转移**       | 判断权移交给不在 AI 生成机制内的主体    | 人审关键决策、规则引擎、物理传感器、事后审计                 |

#### 各类详解

**1. 过程消除**

-   **机制**：重构工作流，从输入到输出只有一条路，路上没有需要判断的分叉口
-   **锚定位置**：判断之前——消除判断点本身
-   **判定点**：不存在，语义操作没有对象
-   **逻辑最彻底性**：判断点不存在，无从绕起
-   **适用边界**：只能在完全可预见的、标准化的子任务中实现。开放性任务无法完全消除判断点

**2. 生成时结构约束**

-   **机制**：外部系统在生成时动态裁剪可行 token 空间。AI 只在裁剪后的空间里采样，外部系统强行屏蔽某些 token
-   **锚定位置**：判断之中——改变判断空间
-   **判定点**：外部系统实时裁剪
-   **特性**：半外部约束——判定不在语义，但也不是后验校验，而是生成时约束
-   **不能归入其他类别的原因**：
    -   不是格式强制（因为不是 AI 自维持）
    -   不是结构判定（因为不是事后判定）
    -   不是过程消除
    -   不是委托转移
-   **典型实例**：function calling schema（OpenAI tools）、constrained decoding（只允许合法 JSON token）、grammar-based decoding（CFG/regex）

**3. 结构判定**

-   **机制**：把判断压缩为二元事实核查，由外部系统执行
-   **锚定位置**：判断之中——改变判断性质
-   **判定点**：外部系统执行二元测试
-   **关键约束条件**：判定行为由外部系统或人执行，AI 只提供被判定物，不参与判定过程。如果核查由 AI 自己执行，就掉回了射程内的逻辑钳制或格式强制
-   **谱系特性**：结构判定是一个谱系，一端是压缩解释空间的模糊事实核查，另一端是彻底消灭解释空间的形式验证
-   **与委托转移的区分维度**：是否允许歧义
    -   结构判定：不允许歧义，必须二元确定
    -   委托转移：允许歧义，依赖判断者的专业经验

**4. 委托转移**

-   **机制**：AI 不判断，判断权移交给一个不在 AI 语义射程内的判断者
-   **锚定位置**：判断之后——把判断权移出行为体
-   **判定点**：外部主体执行判断
-   **委托对象的完整形态**：
    -   委托给人：人的解释不在 AI 的语义射程内；带宽瓶颈是人独有的问题
    -   委托给另一个系统：确定性规则引擎、形式校验器、物理传感器；无带宽瓶颈，可自动化
    -   委托给"未来的自己"：强制输出决策文档，未来审计以承诺为锚点检查一致性；时间维度上的委托

---

### 2.3 约束强度梯度（从弱到强）

基于上述完整分类，约束强度从弱到强排列如下：

**射程内（概率控制）：**

1.  分布层（最弱）——调整角色和价值观的采样分布
2.  注意力层——在决策点提醒规则
3.  轨迹层——约束生成路径结构

**射程外（空间裁剪）：**

4.  结构判定（后验）——事后二元测试
5.  生成时结构约束——事中裁剪 token 空间
6.  委托转移——判断权外移
7.  过程消除——判断点不存在（最强）

**这条强度梯度的工程意义**：它直接指导"什么时候必须出射程"——当风险等级要求 hard guarantee 时，必须从射程内迁到射程外。

## 三、判定权归属——射程内/外的唯一分界线

经过系统化讨论，我们确认了划清射程内/外边界的核心标准：

| 判定执行者                | 分类   |
| ------------------------- | ------ |
| LLM 自身（语义/生成机制） | 射程内 |
| 外部系统（代码/人/物理）  | 射程外 |

**更精确的表述**：  
判定权归属 = 唯一分界线。射程内/外的定义性特征不在于"约束写在哪里"，而在于**"违规是否由模型自身的生成机制来判定"**。

# 第二部分：三层结构——约束来源的正交维度

## 一、三层结构的提出

在确认了射程内/外的完整分类后，我们识别出还有一个维度没有被框架覆盖：**约束从哪里来？** 射程内/外回答的是"约束如何生效"，而约束来源回答的是"生效的东西从哪里来"。

这引出了三层结构：

| 层         | 本质                     | 可设计性                 | 示例                         |
| ---------- | ------------------------ | ------------------------ | ---------------------------- |
| **规约层** | 意图声明                 | 完全可设计               | 行为准则、角色设定、任务指令 |
| **控制层** | 生效机制                 | 完全可设计               | 约束基元、概率控制、空间裁剪 |
| **基线层** | 模型出厂自带的结构性偏差 | 不可设计，只能认知和应对 | 训练偏差、对齐税、架构偏置   |

## 二、三层详解

### 2.1 规约层（Specification Layer）

-   **定义**：所有用语言表达的"应该"
-   **内容**：行为准则（"模型应诚实"）、角色设定、价值观陈述（分布内化的素材）、任务指令（注意力提醒的素材）、输出格式描述（格式强制的素材）
-   **本质**：它是约束的**图纸和意图**，不是约束本身
-   **与控制层的关系**：规约层定义意图，控制层产生效力。规约必须被实例化成控制层的完整约束基元，才能从"应该"变成"必然"

### 2.2 控制层（Control Layer）

-   **定义**：所有能产生控制力的"必然"
-   **内容**：射程内的概率控制（分布层、注意力层、轨迹层的完整基元）、射程外的空间裁剪（过程消除、生成时结构约束、结构判定、委托转移的完整基元）
-   **本质**：它是约束基元的部署层，产生实际控制力
-   **最小单元**：约束基元（作用域-判定器-响应）

### 2.3 基线层（Baseline Layer）

-   **定义**：模型出厂自带的、不可通过设计消除的行为倾向
-   **内容**：
    -   **训练数据分布偏差**：模型在预训练阶段吸收的统计规律，某些输出天然更可能被采样
    -   **对齐税**：RLHF 等对齐训练植入的偏好（过度拒绝、谄媚倾向、避免冲突）
    -   **架构偏置**：注意力机制对位置的偏好、序列长度对质量的影响
-   **本质**：它是约束设计的**默认值**，在你设计任何规约之前已经存在，在你部署任何控制之前已经在生效
-   **工程意义**：
    -   如果不认识这一层，会白白在规约和控制上反复调参，永远找不到病因
    -   基线层定义了出厂默认分布；射程内概率控制本质上是在这个默认分布上做偏移
    -   射程外空间裁剪，裁剪的是基线层和控制层共同定义的空间

---

## 三、三层结构与射程内/外的正交关系

三层结构与射程内/外是两个正交的维度：

-   **地基（射程内/外）回答**：约束如何生效？（概率控制 vs 空间裁剪）
-   **三层结构回答**：生效的东西从哪里来？（设计意图 vs 部署机制 vs 出厂默认）

正交关系表：

|                        | 射程内（概率控制）              | 射程外（空间裁剪）              |
| ---------------------- | ------------------------------- | ------------------------------- |
| **规约层**（设计意图） | "模型应诚实"这条指令            | "输出绝不能包含 SID"这条需求    |
| **控制层**（部署机制） | 注意力提醒基元（指令+模型自检） | 关键词熔断基元（正则+阻断响应） |
| **基线层**（出厂自带） | 训练数据里礼貌比粗鲁概率高      | 模型根本不知道你的内部 API 端点 |

**这个正交模型是约束行为的分析地基**（可作为分析地基，但不能作为预测地基——它没有建模跨层耦合的强度、基线层的精确参数、组合交互的非线性）：

-   **能做到的**：给任何约束行为提供完整分类学坐标；失效发生后系统性地回溯根因；设计新约束时提供决策框架；识别出"缺了什么"
-   **不能做到的**：预测约束配置的实际效果强度；预测多约束组合的涌现行为；取代实际测试

**失败诊断的四个锚点**：

1.  是规约层写错了意图？
2.  是控制层没选对判定器或响应？
3.  还是基线层的出厂偏差在暗中主导行为，根本不理会上两层？
4.  规约到控制的转化是否完整？（是否有规约没有被实例化为控制层基元）

# 第三部分：约束基元——约束的最小完整单元

## 一、约束基元的定义与三要素

### 1.1 为什么需要这个概念

在分类学意义上，不存在朴素如化学元素般不可分割的"原子约束"。但存在一个由三个刚性要素构成的**约束基元**，它才是约束设计的最小完整单元。不能只丢出一个词——任何一条能实际生效的约束，必须同时包含这三个要素。

### 1.2 三要素定义

| 要素                  | 定义                                     | 典型形式                                                     | 示例                                                         |
| --------------------- | ---------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **作用域（Scope）**   | 约束在约束什么？界定了约束生效的精确空间 | 输出类型、上下文条件、时间窗口                               | "所有对外输出的字符串"；"代码生成域"；"医疗场景下的回答"     |
| **判定器（Arbiter）** | 如何判定违规？违规事实如何被建立         | 代码中的正则表达式、外部分类器、人类审查员、constrained decoder | `/\b\d{3}-\d{2}-\d{4}\b/`；"输出是否包含 SID 格式的字符串"   |
| **响应（Response）**  | 违规后会发生什么？违规事实产生的必然后果 | 阻断输出并返回错误信息、标记为待审、记录日志并告警、重写违规部分 | "阻断输出，返回`[输出已拦截]`"；"附加`[ESCALATE]`标记并送入审核队列" |

### 1.3 为什么三者缺一不可

三者缺一个，闭环就断开，不能独立产生控制力，只是一条**规约（Specification）**，而非**约束（Constraint）**。

-   缺少**作用域**：约束不知道在什么情况下生效，无法被实际触发
-   缺少**判定器**："违规"只是模糊的语义概念，不是可触发行动的事实
-   缺少**响应**：判定没有工程意义，约束形同虚设

### 1.4 规约层与控制层的正式分离

基于约束基元的概念，我们可以精确区分：

| 未满足三要素的结构       | 缺少的要素       | 实际位置                                                     |
| ------------------------ | ---------------- | ------------------------------------------------------------ |
| 行为准则（"模型应诚实"） | 缺判定器、缺响应 | **规约层**——语义指导的意图声明                               |
| 知识库中的权威列表       | 缺作用域、缺响应 | **规约层**——数据，当被外部判定器使用时才成为控制层依赖项     |
| "请仔细检查输出"指令     | 缺作用域、缺响应 | **规约层**——注意力提醒/轨迹层语义提示，一个动作而非完整控制闭环 |
| 没有违规响应的熔断规则   | 缺响应           | **规约层**——未被激活的判定条件，形同虚设                     |

**核心推论**：  

> 规约层是用语言表达的意图，天然在射程之内，可被重新解释。  
> 控制层（完整约束基元）才是真正的生效机制，其生效位置取决于判定器放在哪里——在生成机制内就是射程内，在外部系统就是射程外。  
> 规约层到控制层的距离，就是"应该"到"必然"的距离。

---

## 二、基线层——被遗忘的约束来源

### 2.1 为什么三层结构不能仅包含规约层和控制层

规约层和控制层容纳了所有可以被**设计**出来的运行时行为限制。但还有一类东西，既有约束效果，又不属于这两层：**模型内在的结构性偏差**。

-   它在你设计任何规约之前就已经存在
-   它在你部署任何控制之前已经在生效
-   你不能"设计"它，只能"利用"或"对抗"它

### 2.2 基线层的具体内容

**训练数据分布偏差**：

-   模型在预训练阶段吸收的统计规律
-   某些输出在训练分布中天然更可能被采样，某些天然几乎不可能
-   这不是你设计的角色设定在起作用，而是模型本身的"出厂默认分布"在起作用

**对齐税（Alignment Tax）**：

-   RLHF 等对齐训练植入的偏好
-   过度拒绝、谄媚倾向、避免冲突
-   这些不是你的行为准则定义的规约，但在运行域产生了真实的约束效果

**架构偏置**：

-   模型结构本身带来的倾向
-   注意力机制对位置的偏好、序列长度对质量的影响
-   这些不是控制层设计的，但裁剪了可能输出空间

---

## 三、完整体系的启发性问题

这个框架不是一个静态分类，它应该能够启发我们问出对的问题。

### 3.1 设计前：应该问什么？

1.  **基线层分析**：在开始设计任何约束之前，先问：这个模型"天生"就会做什么？我在对抗什么？我利用什么？如果不加任何约束，它在目标场景下的自然行为是什么？

2.  **规约层设计**：从需求和基线出发，需要的"意图"是哪些？把行为准则写清楚。但不自欺——知道这一步只是写"应该"，不是建"必然"。

3.  **控制层部署**：把规约实例化为约束基元。作用域？判定器（射程内或外）？响应？

### 3.2 失败诊断时：应该问什么？

1.  **规约坏了吗？** 意图定义有歧义或自相矛盾？规约没对齐真正的需求？

2.  **控制坏了吗？** 判定器类型选错了（概率控制 vs 空间裁剪）？判定器实现有 bug？响应太弱无法阻断风险？

3.  **基线在作祟吗？** 模型出厂偏差在暗中抵消规约和控制的效果？对齐税导致的过度行为？

4.  **规约→控制转化完整吗？** 某条规约没有被实例化为控制层的任何基元？

5.  **规约→控制转化保真吗？** 规约的意图在实例化时被偷换了？（如"保持诚实"被窄化为"不输出虚假信息"，丢了"不刻意误导"）

6.  **跨多个基元的组合有问题吗？** 基元 A 的响应截断了基元 B 的判定链？几个基元在重叠域里相互冲突？

---

# 第四部分：工程实践

## 一、需求如何转化为约束的表达

这是一个从"模糊意图"到"精确机制"的翻译过程。已有的地基（规约层、控制层、约束基元）为这个翻译提供了完整的工具箱。

需求转化为约束的表达，本质上就是**将一条自然语言的需求，逐步精炼为约束基元三要素（作用域-判定器-响应）的过程**。

### 1.1 常规需求的转化

分为四个递进的阶段：

#### 阶段一：需求解析——提取核心冲突与风险

拿到一条需求（如"AI 不能泄露用户隐私"），第一步不是去想怎么约束，而是去解析这条需求背后真正的问题。

**操作方法**：对需求进行**风险拆解**。

-   **找出核心冲突**：这条需求试图阻止什么？
-   **拆解违规路径**：AI 有哪些方式可能违反这条需求？（直接泄露、间接推断、上下文泄露、元数据泄露等）
-   **定义违规空间**：基于上述路径，将模糊的需求转化为具体的、可操作的违规类别

**产出**：一份**违规模式清单**，而非直接开始设计约束。

---

#### 阶段二：规约化——从风险清单到规约声明

基于违规模式清单，为每一种模式撰写一条独立的**规约声明**。

**规约声明格式**：  

> "针对[违规模式 X]，系统应当[实现什么状态]。"

**举例**（针对直接泄露）：  
"系统应确保所有外部输出都不包含可识别的个人身份信息实体。"

规约声明的特点：仍然是自然语言，是意图，没有约束力。但它把模糊的需求切分成了清晰的、可分别处理的目标。

---

#### 阶段三：控制化——从规约声明到约束基元设计草案

这是翻译过程的核心。针对每一条规约声明，设计一个或多个**约束基元草案**，回答三个问题：

-   作用域在哪？
-   判定器长什么样？
-   响应做什么？

同时，做出整个转化过程中最关键的设计决策：**把判定器放在射程内还是射程外？**

**决策依据**：

| 判定器位置         | 适用场景                                             | 代价                 |
| ------------------ | ---------------------------------------------------- | -------------------- |
| 射程内（概率控制） | 高频、低风险、难以精确定义、需模型"理解"上下文的违规 | 永远有非零的失效概率 |
| 射程外（空间裁剪） | 中低频、高风险、可被明确定义和捕获的违规模式         | 误伤正常输出的可能   |

**示例**：以"直接泄露个人身份信息"为例，从射程内到射程外的多个方案：

| 方案类型                          | 作用域                       | 判定器                               | 响应                               | 适用性                                                 |
| :-------------------------------- | :--------------------------- | :----------------------------------- | :--------------------------------- | :----------------------------------------------------- |
| 射程内方案（语义兜底）            | AI 所有外部输出              | 模型自身（轨迹层自检提示）           | 模型自身（要求重写输出）           | 能处理非格式化隐私，但概率性，防不住对抗性攻击         |
| 射程外方案 A（正则熔断）          | AI 所有外部输出字符串        | 外部正则引擎                         | 外部系统阻断输出，返回默认错误提示 | 对格式化隐私绝对有效；误伤率高；无法应对非格式化隐私   |
| 射程外方案 B（精炼熔断+实体识别） | AI 所有外部输出              | 外部 NLP 模型/云 API（敏感实体识别） | 外部系统阻断并告警                 | 结合语义理解与硬性保证；成本高，有延迟，引入新外部依赖 |
| 射程外方案 C（委托转移）          | 所有带`[ESCALATE]`标记的输出 | 人类审计员                           | 人决定放行、修改或全局熔断         | 是模糊、需复杂判断的最终防线；有带宽瓶颈，无法实时     |

---

#### 阶段四：评估与组织化——从孤立设计到协同体系

草案出来后，进行评估、筛选和组合。

**有效性评估**：回到阶段一的违规模式清单，逐条问：这个约束基元能百分之百覆盖这条违规路径吗？漏网的概率有多大？

**副作用评估**：这个约束基元会误伤多少正常输出？

**组合与分层**：基于评估，将多个草案组合成一个纵深防御体系。

**组织关系审查**：审查层级、域间、时序、强度梯度，确保不是散装规则的叠加，而是指挥有序、边界清晰、时序正确、软硬搭配的架构。

---

### 1.2 抽象需求的转化

抽象需求之所以难处理，是因为它混淆了"目的"和"手段"。拆分它的核心是**先澄清问题，再发散场景，最后收敛为具体的违规定义**。这在已有框架上增加了一个前置的"需求澄清层"。

整个从抽象需求到约束基元的转化，分为五个递进步骤：

#### 第一步：目的澄清——把诉求还原为问题

抽象需求通常直接跳到结论（如"让 AI 变安全"或"AI 输出要有礼貌"）。第一步反向还原为它想解决的底层问题。

**操作方法**：对原始需求进行三次连续追问。

1.  **第一次追问"为什么"**：提出这条需求是想避免什么？在什么场景下发生了什么你不想要的结果？
2.  **第二次追问"这是什么"**：将回答中的模糊词汇拆开。追问"风险"、"有害"在当前任务的具体定义。
3.  **第三次追问"怎么发生"**：追问典型场景。这种"有害"具体是怎样产生的？用户如何交互，模型如何回应？

---

#### 第二步：场景发散——从单点案例到全景覆盖

通过追问得到一两个具体场景后，不要马上开始设计约束。先穷举所有可能触发同一底层问题的场景。

**从五个固定维度横向展开**：

1.  **用户维度**：哪些不同类型的用户会触发它（恶意用户、小白用户、好奇测试者）？
2.  **输入维度**：什么样的输入会触发它（直接的、间接的、多轮对话里慢慢诱导的）？
3.  **输出维度**：违规的输出有哪几种形态（明文、代码、暗示、引用）？
4.  **上下文维度**：在什么上下文里更容易发生（长对话末尾、角色扮演中、系统刚报错后）？
5.  **任务维度**：只在某些特定任务中发生，还是所有任务都可能？

**产出**：一张完整的威胁地图。

---

#### 第三步：维度化——将场景浓缩为关键维度

场景太多无法逐一管理。把所有场景抽象成几个正交的分类维度，建立思考框架。

**示例**：对于"不要给出危险的操作建议"：

-   **危险领域**：物理伤害、化学伤害、网络安全攻击、金融诈骗...
-   **建议形式**：详细步骤、概念性指导、工具推荐、来源暗示...
-   **用户意图**：明显恶意、模糊测试、无知且没有意识到危险...
-   **上下文**：角色扮演中还是严肃的技术问答中？

这个维度框架建立后，任何新的场景都可被迅速归入。

---

#### 第四步：违规定义——从维度框架到可操作的具体规则

基于维度框架，把每个维度交叉出的高概率、高风险区域，定义成可操作的违规规则。

**违规定义格式**：  

> "对于[领域 A] × [形式 B] × [意图 C]的场景，AI 的输出如果包含[特征 X]，则构成违规。"

此时，需求已经完全具体化，可无缝衔接到规约化和控制化阶段。

---

#### 第五步：场景差异化——根据关联程度选择合适的控制策略

一个"宽"的抽象需求，最终可能被拆解成多个"窄"的违规定义，各自匹配不同的控制策略：

-   **可精确锁定部分** → 关键词熔断（射程外结构判定），漏网率为零的绝对拦截
-   **需语义理解部分** → 强角色设定 + 严格的射程内自检指令
-   **高对抗性部分** → 专门的射程外敏感内容识别模型作为第二道防线
-   **最模糊的剩余部分** → 触发委托转移，标记`[ESCALATE]`，交人工审核

**核心价值**：把一个宏大的、无法直接操作的抽象诉求，分解为：一份问题定义、一张场景威胁地图、一个维度分析框架、多个具体违规模式定义，以及最终为这些模式量身定制的、分层分类的控制策略组合。

---

## 二、多约束组合的效力分析

多个约束同时作用时，无法精确预测单一维度的数值，但可以系统性地分析约束之间的**结构关系**。这种结构关系决定了组合后的整体效力是增强、削弱还是内耗。

**核心原则**：不试图预测精确数值，转而分析约束之间的交互关系。这是一个设计审查问题，而非计算问题。

### 2.1 第一步：层级对齐检查

**目标**：确保约束的意图不冲突，力量不内耗。  
**方法**：遍历正交模型，从基线层到规约层再到控制层。

#### 检查 1：基线层 vs 规约层

-   **检查点**：你的设计意图与模型的出厂默认倾向，是同向还是对抗？
-   **分析**：
    -   **同向增益**：规约强化已有倾向，效力顺滑且强大
    -   **对抗衰减**：规约对抗强大基线，仅靠射程内大概率失败，必须引入射程外空间裁剪
-   **结论**：识别所有"对抗"关系，它们是高风险点，需优先配置射程外约束

#### 检查 2：规约层 vs 控制层

-   **检查点**：你的设计意图，是否都有相应的控制层基元来保障？
-   **分析**：如果某个重要规约意图在控制层只有一个射程内基元，没有射程外兜底，那这个意图的保障是脆弱的
-   **结论**：识别所有缺乏射程外兜底的重要意图，这些是新约束的插入点

#### 检查 3：控制层内部——射程内 vs 射程外

-   **检查点**：射程内的概率控制和射程外的空间裁剪之间，是互补还是抵消？
-   **分析**：
    -   **互补（兜底）**：射程内处理高频低风险偏离，射程外处理低频高风险灾难——结构健康
    -   **冲突**：射程内降低触碰概率的同时，射程外熔断阈值过低且范围过广，导致正常相邻输出被频繁误伤
-   **结论**：审查所有在同一作用域上同时存在射程内和射程外约束的配置，检查触发阈值是否相互适配

---

### 2.2 第二步：交互效力映射

**目标**：分析约束基元之间的直接互动。  
**三种主要互动关系**：

| 关系类型                | 定义                                           | 效力影响               | 识别方式与分析                                               |
| :---------------------- | :--------------------------------------------- | :--------------------- | :----------------------------------------------------------- |
| **共振（Resonance）**   | 多个基元在同一或不同层级上，指向同一种合规状态 | **增强（1+1>2）**      | 识别：同时在分布层、注意力层、轨迹层部署同一约束意图。分析：叠加增强合规权重，大幅降低无意偏离概率。但对对抗性攻击仍脆弱——一旦找到一种语义重新解释，所有三层可能同时被绕过 |
| **竞争（Competition）** | 多个射程内基元争夺有限的注意力或分布空间       | **削弱（Trade-off）**  | 识别：任务指令中有两条同等权重的显式规则（如"极度简练"和"穷尽说明"）。分析：检查是否存在内部矛盾，为它们设立优先级，否则产出会摇摆不定 |
| **接力（Relay）**       | 一个基元的失效或局限性，被另一个基元覆盖和补救 | **兜底（Safety net）** | 识别：格式强制（射程内轨迹层）之后，紧跟着外部语法校验器（射程外结构判定）。分析：接力关系越多，系统越强壮。审查整个约束链条，看关键高风险判定点前是否都有前置接力环节 |

---

### 2.3 第三步：动态降级路径分析

**目标**：预判当一个核心约束在压力下失效时，整个约束体系如何响应。  
**方法**：基于递进失效链思想，为每一个核心约束设计一个"失效剧本"。

1.  **选定一个核心约束**（如"不能泄露隐私"）
2.  **假设其第一层防御（射程内注意力提醒）已失效**：模型开始响应诱导时，会怎么做？
3.  **追踪系统响应**：
    -   第二层防御（射程外生成时约束）会触发吗？
    -   如果被绕过，第三层防御（射程外后验结构判定）会触发吗？
    -   如果所有自动化防御都失效，最后防线（委托转移）能留下事后追溯的痕迹吗？

**评判标准**：  
降级只有一步 → 脆弱  
降级步骤清晰、且每一步都独立生效 → 强壮

---

## 三、约束的组织关系分析

约束作为一个体系如何架构和管理，需要四维视角。

### 3.1 维度一：层级关系——约束的"指挥链"

约束不是平行排列的，它们之间有明确的优先级和支配关系。

**三种层级结构**：

| 类型         | 定义                               | 举例                                          | 组织意义                                   |
| ------------ | ---------------------------------- | --------------------------------------------- | ------------------------------------------ |
| **上位约束** | 不可被任何其他约束覆盖的宪法级规则 | "绝不泄露系统 prompt"、"绝不对人造成物理伤害" | 硬上限，其他所有约束必须让步，是最终裁定者 |
| **同级约束** | 同等优先级，冲突需上下文判定       | "回答要简洁" vs "回答要充分"                  | 冲突靠上下文权重解决，需定义冲突仲裁规则   |
| **下位约束** | 可被上位约束覆盖的操作性规则       | "输出必须使用 markdown 格式"                  | 有条件生效——当与安全性冲突时，放弃格式     |

**分析方法**：把所有约束列出来，两两之间问——如果它们冲突，谁让步？让步的那个是下位约束。把让步链画出来，形成**约束层级树**。

---

### 3.2 维度二：域间关系——约束的"管辖边界"

每个约束都有一个作用域。分析组织关系就是搞清楚这些域的边界关系。

**四种域间关系**：

| 关系     | 定义                             | 风险                                                         | 分析方法                                                     |
| -------- | -------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **包含** | 约束 A 的作用域完全在约束 B 之内 | 外层约束可能无意识地覆盖或削弱内层约束；外层熔断可能误伤内层精细行为 | 检查：当外层约束触发响应时，内层约束是否还有机会生效？响应链是否被截断？ |
| **重叠** | 两个约束的作用域部分交叉         | 交叉区是冲突高发区。两个约束的判定器和响应必须被协调，否则出现矛盾指令 | 识别出所有重叠域，为每个重叠域显式指定主导约束。未指定的重叠域是系统风险 |
| **邻接** | 两个约束的作用域相邻但不重叠     | 二者之间可能存在未被覆盖的盲区——约束的裂谷，模型可能从域 A 滑入域 B 的缝隙不受控制 | 检查邻接线上是否有覆盖率缺口，反复测试邻接线上的模型生成是否摆动 |
| **全异** | 两个约束的作用域完全不相关       | 需注意间接影响——一个域内约束触发的高频响应可能影响模型在另一个域内的生成状态 | 无明显风险时无需特殊处理，但需监控跨域污染                   |

**分析方法**：画维恩图，把约束按作用域放入。检查每一个重叠区、每一条邻接线。没画出来的盲区，就是约束体系的空白地带。

---

### 3.3 维度三：时序关系——约束的"生效节奏"

约束不是同时生效的。它们有先后、有前置依赖、有触发条件。

**三种时序关系**：

| 关系     | 定义                                                | 举例                                                         | 组织意义                                                     |
| -------- | --------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **前置** | 约束 A 必须先于约束 B 部署或生效，B 依赖于 A 的结果 | constrained decoding 必须在语义自检之后生效                  | 前置关系颠倒导致逻辑链断裂。确保依赖链的前后顺序在系统构建中正确映射 |
| **并发** | 两个约束同时生效，作用于同一生成步骤                | 输出 token 时同时通过 logit_bias 屏蔽和 constrained_decoding 裁剪 | 并发约束的交互是动态的。分析：它们的判定器是否在同一时刻给出矛盾指令？ |
| **触发** | 约束 B 只在约束 A 触发特定条件后才激活              | 递归反思提示只在检测到第一次生成结果包含高风险词后才激活     | 触发关系节省计算成本。但必须分析：被依赖的约束 A 是否会漏检？漏检率决定整个触发链的可靠性 |

**分析方法**：画出单次生成任务中的**约束时序线**——什么时候哪个约束被激活、什么时候退出、什么时候判定结果被使用。时序线上的缺口和堆积点，都需仔细分析是否构成安全问题或性能瓶颈。

---

### 3.4 维度四：强度梯度——约束的"力量分布"

约束体系不应是扁平的"所有约束同等强力"的结构，而应有从软到硬、从低成本到高成本的梯度。

**组织原则**：

-   **默认层是概率控制**（低成本、高覆盖、低保证）：射程内约束覆盖日常高概率偏离
-   **中间层是半硬性约束**（中等成本、中保证）：生成时进行 token 级裁剪
-   **最高层是硬性空间裁剪**（高成本、高保证、可能误伤）：部署后验结构判定和委托转移

**分析方法**：

-   **识别薄弱区**：是否存在高风险违规模式，只被软性约束覆盖、没有被任何硬性约束兜底？
-   **识别过度约束区**：是否为了防止低危害违规，部署了多个高成本的刚性约束，导致合规输出频繁受阻？

---

## 四、重复约束的分析

同一约束在不同位置出现，逻辑上只有三种情况：

### 4.1 三种基本关系

**1. 等价（不同表述，同一规则）**

-   两者指向同一行为集合，是**同义复现**
-   **位置意义**：在系统指令头和尾各出现一次 = 强化；在同一条任务指令里出现两次 = 赘余或措辞失误

**2. 子集（一个比另一个更严格）**

-   通用规则："保护用户隐私" → 特定规则："绝不输出身份证号、手机号、家庭住址"
-   **这才是工程上最重要的情况：子集关系不是重复，是特化。** 上层的宽泛约束是兜底，下层精确锚定是熔断锚点

**3. 引用（一处指认另一处为权威）**

-   "请遵守上述《核心行为准则》"
-   引用本身不产生新约束，它建立的是约束之间的优先级或权威关系

---

### 4.2 重复产生的三种效应

| 效应         | 机制                                                         | 适用条件                                                 |
| ------------ | ------------------------------------------------------------ | -------------------------------------------------------- |
| **共振增强** | 等价或子集约束在多个位置复现，规则相关 token 的注意力权重累加 | 对无意偏离有效。对模型已构建合理化叙事来绕开的情况无效   |
| **稀释风险** | 宽泛规则的精确子集从解释空间里抽出，做成硬性锁定             | 子集关系最重要的工程价值                                 |
| **冲突潜伏** | 不同表述引入不同的解释边界，在特定输入下互相矛盾             | 最危险的坑：不是重复本身有问题，而是不同表述导致边界矛盾 |

---

### 4.3 反效模式——需要特别警惕的陷阱

**陷阱一：重子集弱化原集（虹吸效应）**

-   **机制**：当你在某处部署了高度具体的子集约束，模型在满足这个硬性要求后，在其他相邻但未被锁定的维度上系统性放松
-   **本质**：注意力分配的零和博弈——精确聚焦在子集上，反而削弱了对原集其他部分的注意力分配
-   **诊断标准**：精确锁定某些违规模式的同时，相邻同属原集的违规模式频率反而上升
-   **对策**：在上升维度上部署对应的精确锚定，或将原集约束从射程内语义指导升级为射程外结构判定

**陷阱二：引用/等价强化错位**

-   **情况 A（指向模糊 → 注意力稀释）**：等价复述用了更模糊的措辞，抽象声明和具体指令同时存在时，注意力被分流到抽象维度，实际生效精确度下降
-   **情况 B（指向已被推理覆盖 → 空转）**：引用触发冗余的注意力回溯，抽离对当前任务的关注去重新关注已生效规则，增加噪音和偏差
-   **诊断方法**：逐一审查每条引用和等价复述，问两个问题：指向是有实质内容的规则还是抽象口号？发生位置是决策点之前（提醒）还是之后（干扰）？

---

### 4.4 处理决策树

| 关系类型        | 处理方式                                                     |
| --------------- | ------------------------------------------------------------ |
| 等价 + 蓄意共振 | 保留，标注"蓄意共振，修改需同步"                             |
| 等价 + 无意赘余 | 合并，减少维护成本和措辞之间的语义缝隙                       |
| 子集关系        | 确认为特化，不合并。宽泛约束保留为意图声明（规约层），子集约束保留为硬性锚点（控制层）。**必须显式标注二者关系**："本规则是《A》关于 X 的特化，判定时本规则优先" |
| 引用关系        | 检查引用链完备性。每一条引用必须能追溯到有实际内容的约束。悬空引用是断链，必须修复。循环引用代表逻辑悖论，必须断开 |
| 任何关系类型    | **必须检查冲突**。存在冲突 → 立即为该冲突场景设定优先级规则，并显式写入引用声明中。不存在冲突 → 记录为"已验证无冲突" |

---

## 五、约束膨胀的治理

当需求很多，转化后的总约束表达又很长时，可能造成系统提示词暴涨、模型忽略或在约束之间迷路。这就是**约束膨胀**。

**总原则**：不要在同一个平面上堆砌，用立体结构来代替平面铺陈。

### 5.1 方法一：规约与控制分离——最根本的一刀

-   **问题根源**：把所有东西（意图、规则、熔断条件、响应方式）全部用自然语言塞进系统提示词
-   **对策**：
    -   **系统提示词只保留规约层的意图和少量最核心的注意力提醒**。如："你在医疗场景下应保持高度谨慎，并严格遵守《医疗约束集 v2.1》"
    -   **控制层全部外化**：正则熔断、关键词列表、constrained decoding 语法定义全部放在代码或配置文件中，不进入模型上下文窗口
-   **收效**：模型上下文负担大幅降低，控制层规则可从几百条扩展至上万条而不占注意力

---

### 5.2 方法二：动态分层加载——只给模型当下需要的约束

核心是让约束**按需出现**，而不是一次性全部加载。

| 组织关系      | 动态加载策略                 | 示例                                                         |
| :------------ | :--------------------------- | :----------------------------------------------------------- |
| 域间包含/重叠 | 进入某作用域时才注入该域约束 | 进入"代码生成域"才注入代码安全约束；闲聊域中这些约束不出现   |
| 时序前置      | 触发前置条件后才加载后续约束 | 用户问到医疗→注入医疗约束集；切换到法律→切换法律约束集       |
| 强度梯度      | 风险等级升级时逐级追加约束   | 日常对话只挂基础礼貌约束；检测到高风险输入→动态追加强硬拒绝模板和关键词熔断 |

这样，模型在任何时刻看到的都是当前任务域下的**精简约束子集**。

---

### 5.3 方法三：等价合并与分层引用——在约束本身做优化

-   **等价合并**：审查所有约束，识别同一约束基元的多个等价表述，只保留一个最清晰完整的版本
-   **分层引用**：系统提示中只植入**索引约束**——轻量级注意力提醒，不陈述详细规则，而是精确引用到外部规则集并定义触发时机。示例："当输出涉及外部数据时，执行《外部数据引用策略》"
-   **构建约束树**：总约束集组织成树形结构，主干是少量顶层宪法级规则，分支是各域具体规则。模型上下文始终只保留主干和当前激活的分支

---

### 5.4 方法四：约束编译——将规则集转译成模型"原生"的约束形式

**核心思想**：如果约束实在太多太长，就不要让它以文本形式存在。将大量硬性规则"编译"成模型更底层、不占用上下文窗口的控制机制。

| 规约类型         | 编译目标                                                     |
| ---------------- | ------------------------------------------------------------ |
| 正则关键词       | logit_bias 或 constrained_decoding 语法——编译为生成时绝不会被采样的 token 集合 |
| 模糊语义规则     | 外部分类器——微调轻量级分类模型，在输出后做判定               |
| 角色设定与价值观 | 模型微调——永久的、高频的约束通过微调内化到基线层，从根本上消除占用上下文窗口的需求 |

---

### 5.5 最终理想状态

模型看到的是一个精简至核心意图和关键索引的系统提示：

-   按需加载当前域的详细指令
-   大部分硬性和重复性规则要么被编译进生成机制，要么由外部引擎后验兜底
-   约束的总量可以无限膨胀，但模型的直接认知负担被压到最低

---

## 六、同一作用空间内并行约束的优先级设计

当多个约束在同一作用域内并行运作时，必须为它们冲突的可能性提前制定仲裁法则。

### 6.1 源优先级原则

**原则一：可行性压制原则**

> **射程外约束 > 射程内约束**

射程外约束裁剪的是"可行性本身"，射程内约束只是调整概率。当射程外熔断器和射程内注意力提醒发生冲突时，熔断器无条件胜出。

**原则二：后果等级原则**

> **安全硬约束 > 合规约束 > 质量/风格约束**

-   **安全硬约束**：违规将造成伤害
-   **合规约束**：违规不直接造成伤害，但会造成运营事故
-   **质量/风格约束**：违规不影响实质，只影响体验

**原则三：判定权外移优先原则**

> **外部判定 > AI 自判定**

外部判定的结果是确定事实，AI 自判定的结果是概率抽样。

---

### 6.2 裁决冲突的元法则

**法则 1：横跨优先**——先看"射程"，射程外约束直接胜出（参见原则一）

**法则 2：后果称重**——二者都在射程内或都在射程外时，安全 > 合规 > 质量（参见原则二）

**法则 3：权属裁定**——后果等级相同时，外部判定 > AI 自判定（参见原则三）

**法则 4：特定优先**——若上述全部平等，则子集约束 > 父集约束。因为子集是对父集的特化，表达了更精确的设计意图

**法则 5：显式仲裁**——若精准度层级也分不出胜负，则这两个约束设计上实质矛盾。此时不能自动裁决，必须上升为显式仲裁，设计师必须介入写入明确的优先级规则

---

### 6.3 优先级映射表示例

| 场景 | 冲突约束 A                        | 冲突约束 B                                           | 裁决     | 依据                                        |
| :--- | :-------------------------------- | :--------------------------------------------------- | :------- | :------------------------------------------ |
| C-01 | 字数 ≤200 字（质量，射程内）      | 涉医时穷尽所有风险（安全，射程内）                   | B 胜出   | 原则二（安全>质量）                         |
| C-02 | AI 自检确认无 PII（安全，射程内） | 外部正则引擎阻断疑似 PII（安全，射程外）             | B 胜出   | 原则一（射程外>射程内） + 原则三（外部>AI） |
| C-03 | 绝不输出身份证号（安全，射程外）  | 遵守全局回复模板，模板含用户数据字段（合规，射程外） | A 胜出   | 原则二（安全>合规）                         |
| C-04 | 回答要简洁（质量，射程内）        | 回答要详尽（质量，射程内）                           | 显式仲裁 | 法则 5——设计师必须介入                      |

---

### 6.4 无法自动仲裁的剩余冲突

对于掉到法则 5 的冲突，建立轻量级的人工升级与处置流程：

1.  **标记**：系统在逻辑扫描中发现两条约束完全同级且实质矛盾时，输出`[PRIORITY_CONFLICT]`标记
2.  **升级**：将这对约束连同当前场景记录到专门的任务队列或日志
3.  **处置**：设计师做出裁决，将裁决输入回系统，形成动态优先级补丁，进入映射表，防止同一冲突再次发生

---

## 七、约束设计的总体原则总结

### 7.1 设计原则

1.  **射程外优先**：对于高风险的违规模式，优先考虑射程外空间裁剪，而非依赖射程内的概率控制
2.  **纵深防御**：不依赖单一约束，构建射程内→半外部→射程外的多层接力
3.  **规约控制分离**：系统提示词保留意图，控制逻辑外化到代码和配置
4.  **基元完整性**：每一条约束必须包含完整的"作用域-判定器-响应"三要素
5.  **优先级的预设**：在部署前就按"射程外>射程内、安全>合规>质量、外部>AI"的原则预设好冲突仲裁规则

### 7.2 诊断原则

1.  **失效先查基线**：约束失效时，先问是不是基线的出厂偏差在作祟
2.  **追查规约到控制的转化**：每条重要规约是否都有对应的控制层基元？
3.  **检查组合交互**：多个约束之间是共振、竞争还是接力？是否存在未处理的冲突？
4.  **审查组织完整性**：层级树是否清晰？域间是否有盲区？时序是否正确？强度梯度是否合理？
5.  **警惕反效模式**：子集是否反而削弱了原集？引用是否指向模糊导致注意力稀释？

| 空间裁剪       | 改变可行空间，剪掉违规可行性，可提供 hard guarantee          |
| 分布层         | 射程内最底层，通过改变 P(output \| context) 实现约束         |
| 注意力层       | 射程内中间层，通过改变当前 step 的 token 权重分配实现约束    |
| 轨迹层         | 射程内最高层，通过约束生成序列路径（语义一致性或语法路径）实现约束 |
| 过程消除       | 射程外约束：重构工作流，使需要判断的时刻不存在               |
| 生成时结构约束 | 射程外约束：外部系统在生成时动态裁剪可行 token 空间          |
| 结构判定       | 射程外约束：后验二元事实核查，由外部系统执行                 |
| 委托转移       | 射程外约束：判断权移交给不在 AI 生成机制内的主体             |
| 约束基元       | 约束的最小完整单元，包含作用域、判定器、响应三要素           |
| 作用域         | 约束生效的精确空间界定                                       |
| 判定器         | 违规事实如何被建立                                           |
| 响应           | 违规事实产生后的必然后果                                     |
| 规约层         | 意图声明层，所有用语言表达的"应该"                           |
| 控制层         | 生效机制层，由约束基元构成，产生实际控制力                   |
| 基线层         | 模型出厂自带的结构性偏差，不可设计只能认知和应对             |
| 对齐税         | RLHF 等对齐训练植入的偏好（过度拒绝、谄媚倾向等）            |
| 虹吸效应       | 子集约束精确锚定后，削弱原集其他维度注意力的反效模式         |
| 共振           | 多个约束指向同一合规状态，产生 1+1>2 的增强效果              |
| 竞争           | 多个约束争夺有限注意力或分布空间，互相削弱                   |
| 接力           | 一个约束的失效被另一个约束覆盖和补救，形成纵深防御           |
| 约束膨胀       | 约束数量过多导致系统提示词暴涨、模型忽略或在约束间迷路       |
| 索引约束       | 轻量级注意力提醒，精确引用外部规则集并定义触发时机           |
| 约束编译       | 将文本规则转译成不占用上下文窗口的底层控制机制               |
| 可行性压制     | 优先级原则：射程外约束无条件优先于射程内约束                 |
| 后果等级       | 优先级原则：安全 > 合规 > 质量                               |
| 判定权外移优先 | 优先级原则：外部判定 > AI 自判定                             |
